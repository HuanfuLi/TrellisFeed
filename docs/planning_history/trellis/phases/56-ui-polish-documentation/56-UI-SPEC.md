# Phase 56 — UI Design Contract

> Visual and interaction contract for the Phase 56 UI Polish & Documentation sweep.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

> **PHASE NATURE — read first.** This is NOT a build-a-new-screen phase. Phase 56 is a
> cross-cutting **polish + audit + documentation** sweep over the EXISTING Trellis app
> (React 19 + TypeScript + Vite + Tailwind CSS 4 + Capacitor). There is no new feature
> surface to design. The job of this contract is to **codify the existing inline-style +
> CSS-variable design system into a checkable acceptance contract**, and to define the
> **polish / animation / navigation rubrics** the sweep is measured against. Every token
> below is EXTRACTED from `app/src/index.css` and live `*.tsx` usage — it is the locked
> source of truth, not a new proposal. Do NOT introduce a new palette, typography scale,
> spacing system, or motion language. (Adding new motion is explicitly declined per D-04.)

## Spacing Scale

**Source of truth:** `app/src/index.css` `--space-*` tokens. The contract is the EXISTING scale. The polish sweep checks adherence to these tokens and flags arbitrary off-scale pixel values (e.g. `padding: 13px`, `margin: 7px`) as candidate findings.

| Token | Value | Usage |
|-------|-------|-------|
| `--space-xs` | 4px | Icon gaps, inline padding |
| `--space-sm` | 8px | Compact element spacing |
| `--space-md` | 12px | Tight element spacing (in-system 12, not 16) |
| `--space-lg` | 16px | Default element spacing |
| `--space-xl` | 20px | Card / row padding |
| `--space-2xl` | 24px | Section padding (`--section-gap: 24px`) |
| `--space-3xl` | 32px | Layout gaps / major section breaks |

**Radii:** `--radius: 1.25rem` (20px) · `--radius-xl: 1.75rem` (28px) · `--radius-pill: 9999px`.
**Shadows:** `--shadow-1` (resting), `--shadow-2` (raised), `--shadow-3` (overlay/modal). Use the named tier, never an ad-hoc box-shadow.
**Safe-area / layout:** `--safe-area-top`, `--safe-area-bottom`, `--bottom-nav-height: calc(80px + safe-bottom)`, `--bottom-nav-safe`. Scroll containers must respect these — a polish finding if content is occluded by the nav or notch.

**Exceptions (audit-relevant, do NOT flag):**
- Touch targets: interactive controls (icon buttons, nav taps, Send) target **≥44px** minimum hit area even when the visual glyph is smaller. ChatInput buttons are fixed at 44px (CLAUDE.md ChatInput invariant) — do not shrink.
- The scale is **not** a strict 8-point grid: it intentionally includes 12px and 20px. Treat 4 / 8 / 12 / 16 / 20 / 24 / 32 as the allowed set. Pixel values OUTSIDE this set in a fixable location are candidate findings, subject to operator triage (D-01).

## Typography

**Source of truth:** live `fontSize` / `fontWeight` usage across `src/**/*.tsx`, base `--font-size: 16px`. Trellis uses a `rem`-anchored type scale (most common: `0.875rem` body-small, `0.75rem` caption). The contract codifies the **role buckets**; the polish sweep flags one-off sizes that don't map to a role and inconsistent weight usage for the same role.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display / hero | 2rem–3rem (32–48px) | 700 | 1.2 |
| Heading (screen / section title) | 1.25rem (20px) | 600–700 | 1.25 |
| Body | 1rem (16px) / `0.95rem` | 400 | 1.5 |
| Body-small (dominant) | `0.875rem` (14px) | 400–500 | 1.5 |
| Label / metadata | `0.75rem` (12px) | 500–600 | 1.4 |
| Caption / chip | `0.7rem`–`0.72rem` (~11–12px) | 500 | 1.3 |

**Weights in use:** 400 (normal, `--font-weight-normal`), 500, 600, 700 (and 800 for hero numerics). The system intentionally uses 4 weights; this is NOT a 2-weight system. The audit checks **weight consistency within a role**, not weight-count reduction. Do not collapse the weight set.

**Audit guidance:** body line-height must stay at/near **1.5**, headings **~1.2**. Flag dense text blocks below 1.4 line-height as readability findings. Flag truncation/overflow that clips text (especially es locale, which runs ~20% longer — CLAUDE.md i18n note).

## Color

**Source of truth:** `app/src/index.css` `:root` (light) + `[data-theme="dark"]` overrides. The 60/30/10 split below DESCRIBES the existing palette so the auditor can check contrast and accent discipline — it is not a new palette.

| Role | Value (light) | Usage |
|------|---------------|-------|
| Dominant (60%) | `--surface` `#FFFBF5` / `--background` `#FFFBF5` | App background, surfaces (warm off-white) |
| Secondary (30%) | `--card` `#FFFFFF`, `--surface-variant` `#F5F0E8`, `--surface-container` `#FFF8F0` | Cards, rows, nav, settings panels |
| Accent (10%) | `--primary-40` `#558B2F` (Trellis green) | See reserved-for list |
| Destructive | `--danger` `#E53935` (`--danger-light`, `--danger-dark`) | Destructive actions ONLY |

**Accent (`--primary-40`) reserved for:** primary CTAs / confirm buttons, active nav-tab indicator, primary progress fill (vine/trellis growth), selected/active states, key links. NOT for decorative borders, body text, or every interactive element.

**Secondary accent palette (NOT general-purpose):**
- `--secondary-40` `#FFD54F` (amber) — fruit credits, harvest reward, milestone highlights.
- Node colors `--node-mint / -salmon / -lilac / -peach / -sky` — graph nodes, bento card backgrounds. Semantic to the knowledge-graph + planner garden; do not repurpose for chrome.
- `--accent-coral` / `--accent-lavender` — limited decorative accents.

**Audit guidance:**
- **Dark-mode parity is a first-class check.** Every polish fix must hold in BOTH `[data-theme]` states; flag any hardcoded hex that breaks dark mode (must reference a var). Dark remaps `--primary-40` to `#4CAF50` and folds bento backgrounds to `--card`.
- Contrast: body text on its surface must meet WCAG AA (4.5:1). `--muted-foreground` on `--surface` is the most at-risk pairing — flag low-contrast metadata.
- Destructive red is reserved for delete/clear/prune/destructive-confirm only — flag any decorative use of `--danger`.

## Animation Contract (POLISH-02)

**Stance (D-04, locked): FIX-JANKY-ONLY. Add NO new motion or micro-interactions.** Codifies the operator's "snappy instant beats a janky animation" instinct (2026-04-15 tab-transport revert). The auditor repairs stutter/flicker/dropped-frame in EXISTING animations only.

**Existing motion inventory (34 `@keyframes`, the surfaces the sweep covers):** `fade-in/out`, `slide-up`, `slide-in-left/slide-out-left`, `sub-screen-in/out`, `drill-in-enter/out`, `flashcard-flip-in/flashcard-next`, `node-pop`, `edge-draw`, `confetti-fall`, `fruit-fly`, `aha-pop/aha-pulse`, `glow-pulse/glow-ring/status-glow`, `milestone-pop`, `mic-pulse`, `vineLoadingPulse`, `skeleton-pulse/shimmer`, `toast-in/out`, `bounce`, `spin/btn-spin`, `blink`, `pulse`.

**WebView performance budget (the bar findings are measured against):**
- **Animate compositor-only properties: `transform` and `opacity`.** Flag any keyframe/transition animating `width`, `height`, `top`, `left`, `margin`, `box-shadow`, or `background` on a per-frame path as a jank candidate (these trigger layout/paint on Android Chromium WebView).
- **Target 60fps; budget is no dropped frames on a mid-tier Android device.** No precise FPS instrument is mandated (operator's discretion per D-05) — the auditor uses visible-stutter judgment plus the property-class heuristic above.
- **Respect the load-bearing layout invariants** (see Constraints): NEVER add `transform` / `will-change` / `filter` / `contain` / `perspective` to a `Header` ancestor to "smooth" an animation — that re-breaks the portal-vs-in-tree containing block. Fix jank at the animating element, not by mutating ancestors.
- **Durations** already in use cluster at 150/180/200/250/300ms (UI transitions) and 1–1.5s (ambient loops). Keep transitions in the 150–300ms band; do not lengthen to mask jank.
- `prefers-reduced-motion` honored where present; flag ambient loops (glow/pulse) that ignore it as findings.

**Unsmoothable case (D-05):** when an existing animation cannot hit the budget, the auditor proposes **remove-vs-simplify per animation** with a one-line rationale, and the **operator decides each case during triage** — no blanket remove/keep rule. Default lean: simplify to a `transform`/`opacity`-only version; fall back to instant (snappy) over janky.

## Navigation & Back-Button Contract (POLISH-03)

**Stance (D-06/D-07):** full route-map audit of every route + entry point; **Android hardware/gesture back-button consistency is the TOP concern** (no single known-broken path — special attention everywhere, especially sub-screen Outlets and `settings/` sub-pages).

**Route map (the surface to walk):**
- Swipe-tab roots (always-mounted, instant tab transport): `/home`, `/planner`, `/ask`, `/graph`, `/settings`.
- Sub-screen Outlets (fullscreen overlay, zIndex 50, Header portals to body): `/ask/:id`, `/posts/:id`, `/anchor/:id`, `/cluster/:id`, `/questions/:id`, `/review`, `/podcast`, `/saved`, `/collection/:id` (CollectionDrillIn), `/settings/{ai,content,features,data}`.
- `/onboarding` (gated on `onboardingCompleted`).
- Event-bus-driven navigations (e.g. heal→`/review`, replant→`/posts/anchor-post-{id}`, NEWS/GRAPH events).

**Back-button acceptance rubric (every screen must satisfy):**
1. **Android hardware/gesture back is wired and predictable.** A back press from a sub-screen returns to its logical parent (the screen that opened it), NOT app-exit or a wrong root. Back-handling sites already touched: `App.tsx`, `Header.tsx` (`backTo`), and per-screen handlers — verify they all register/teardown and resolve to the correct parent.
2. **No dead ends.** Every reachable screen has a working way back (Header back-arrow `backTo` AND hardware back agree on destination).
3. **No wrong paths.** Back never lands on a screen the user didn't pass through; deep-link / event-bus entries set a sane back target.
4. **Outlet/Header parity.** Sub-screen Header back-arrow and hardware back go to the SAME destination.
5. **Always-mounted resync on return.** Returning to a swipe-tab root re-reads service state (HomeScreen canonical `[location.pathname]` resync pattern) — flag stale UI on back-return as a finding (CLAUDE.md "always-mounted screens must re-read").
6. **Settings sub-page back** returns to the Settings menu, not the previous swipe tab.
7. **Keyboard-open back** dismisses keyboard cleanly without nav drift (consistent with SwipeTabContainer `onFocusOut` scrollLeft reset).

Findings feed the D-01 hybrid triage; agents fix only operator-approved paths.

## Polish Checklist (POLISH-01) — 6-Pillar Audit Rubric

**Discovery model (D-01, locked):** agent visual audit (gsd-ui-auditor 6-pillar style) produces a SCORED candidate-findings list → operator triages (approve / cut / add) → agents fix ONLY approved items. **Coverage (D-02):** all screens at equal depth — 5 swipe roots + every sub-screen listed in the route map. **Fresh audit (D-03):** from current code; do NOT use the Apr-16 `UI_AUDIT_REPORT.md` as baseline (predates i18n, masonry feed, SQLite, Trellis rework) — archive it.

Each finding scored per pillar; each fix verified against the contract above:

1. **Copywriting** — empty/error/loading states present and clear; CTA labels are verb+noun; no placeholder/lorem; all user-facing strings flow through i18n (`t()`), present in all 4 bundles (en/zh/es/ja); no untranslated hardcoded copy. (Proper nouns stay English — CLAUDE.md i18n list.)
2. **Visual hierarchy** — clear primary/secondary/tertiary emphasis per screen; one primary action per view; consistent card/row treatment; alignment to a shared edge; no orphaned/misaligned elements.
3. **Color** — token-only (no stray hex); accent discipline (reserved-for list); destructive red reserved; dark-mode parity; AA contrast.
4. **Typography** — role-mapped sizes; consistent weight-per-role; line-height ≥1.4 body; no clipped/overflowing text (check es locale length).
5. **Spacing** — `--space-*` token adherence; consistent rhythm; safe-area + bottom-nav respected; no off-scale ad-hoc pixels in fixable spots.
6. **Motion** — see Animation Contract; jank flagged, fix-janky-only, no new motion.

**Feed-tile bias (locked specific):** if the audit touches feed tiles (`InfoFlow.tsx`), bias toward SIMPLIFICATION — fewer per-tile signals, clearer hierarchy — NOT added chips/metadata (feedback_tile_simplicity_preference). Dedicated screens + PostDetail may remain interactive/state-rich.
**No pushy mechanics:** polish must NOT introduce streaks/goals/nudges/scarcity — cosmetic-only (feedback_no_pushy_engagement_mechanics).

## Constraints — Load-Bearing Invariants (MUST NOT REGRESS)

These CLAUDE.md invariants bound every polish/animation/nav fix. A fix that violates any is an automatic FAIL, not a finding.

1. **Header portal-vs-in-tree** — NEVER add `transform` / `will-change` / `filter` / `contain` / `perspective` to any `Header` ancestor. Don't move `Header.tsx` out of the portal-vs-in-tree split. (`tests` enforce; recurred 6+ times.)
2. **Root overflow clip — both axes** — keep `html, body { overflow: hidden }`, the App root `overflowX: 'hidden'`, and the `onFocusOut` `scrollLeft = 0` reset. Never introduce a scroll region at the page root; never rely on body scroll.
3. **SwipeTabContainer resize/keyboard** — keep the `newWidth === screenWidthRef.current` early-return in `resync()` (height-only/keyboard resizes are no-ops); keep `onFocusOut` re-snap. Don't install `@capacitor/keyboard` with `resize:'none'`.
4. **ChatInput flex shrink** — keep `minWidth: 0` on the ChatInput input; don't grow buttons past 44px.
5. **Always-mounted resync** — swipe-tab roots must re-read service state on `[location.pathname]` navigation; don't rely on remount (Capacitor: users can't refresh).
6. **No-refresh assumption** — reactive re-read via event-bus on every mutation, never remount/reload.
7. **One question at a time** — any `AskUserQuestion` during triage contains exactly one question.

## Copywriting Contract

This phase ships no NEW user-facing copy (audit/polish). The CTA/empty/error contract below is the **acceptance bar the audit checks existing copy against**, not new strings to author. Any copy ADDED by a fix must land in all 4 locale bundles.

| Element | Copy contract |
|---------|---------------|
| Primary CTA | Verb + noun, i18n-keyed, accent-colored, one-per-view (e.g. "Review now", "Generate podcast", "Ask"). Flag bare "OK"/"Submit". |
| Empty state heading | Present + specific to the screen (not generic "No data"). Flag missing empty states. |
| Empty state body | One line + a next-step affordance (what to do to fill it). |
| Error state | Problem + recovery path via `toast()` (CONVENTIONS error-handling). Flag silent failures. |
| Destructive confirmation | `ConfirmDialog` with named action + consequence; destructive button in `--danger`. Covers: Clear-All-Data, delete post/session/flashcard, prune/dead-node, force-new-day. |
