# Phase 55.1 Plan 07: Cold-Start First-Response Stall (GAP-C) Summary

Instrumented the first-ask path, measured the dominant cold-start stall (the cold
filter-corpus embed at 6875ms / 89.6% of in-process first-ask cost), and fixed it
with a fire-and-forget boot pre-warm of the filter-corpus embedding cache — moving
the 124-sequential-embed cost off the first-ask path to app boot, with the
filter→chatStream order, RAW-ARGMAX malicious gate, and byte-stable system prompt
all preserved.

## What Was Built

### Task 1 — Instrument + measure (commit `77454448`)
- `app/src/lib/cold-start-profiler.ts`: debug-gated (`trellis_cold_start_profile`)
  `performance.now()` spans, single-shot per process, **no-op when the gate is off**.
- `useQuestions.askStreaming` wrapped (timing ONLY — order unchanged):
  - `(b)` filterQuestion (embedding warm-up + cold corpus embed)
  - `(c)` buildCandidateContextPack (context assembly)
  - `(d)` chatStream TTFT (provider handshake) — flushes the console table on first token.
- `app/scripts/profile-cold-start.mjs`: drives the in-process phase split in the
  `node --test` harness, reading the **real** `filter-corpus.json` entry count (124)
  so the measurement can't drift from the shipped corpus.

### Task 2 — Targeted fix (commit `659c9c06`)
- `prewarmFilterCorpus(embConfig)` added to `filter-corpus.service.ts`: warms the
  SAME `loadCorpusEmbeddings` cache. **Non-blocking** (returns a Promise the render
  path does not await), **single-flight** (`_prewarmInFlight` latch dedups concurrent
  callers + a boot-warm-up-racing-first-ask), **key-absent / offline-safe** (no-op
  when `!embConfig.isConfigured`, swallows embed errors — first ask still works).
- Wired fire-and-forget into the `App.tsx` boot effect (NOT awaited → no first-paint delay).
- `app/tests/state/cold-start-instrumentation.test.mjs`: **executing** invariant via
  the `embedSpy` mock — after pre-warm the first ask pays **0** corpus embeds; pre-warm
  no-ops when unconfigured; single-flight embeds the corpus exactly once.

## Cold-Start Phase Split (Task 1 measurement artifact)

`node --test scripts/profile-cold-start.mjs` — in-process first-ask split
(corpus entries = 124, representative embed = 55ms/call):

| Phase | ms | % | Notes |
|-------|----|----|-------|
| (a) db hydration | 0 | 0% | Already AWAITED at boot in `App.tsx hydrateAllFromSQLite()` BEFORE first render — off the first-ask path by design. NOT the stall. |
| **(b) filterQuestion (cold corpus + query embed)** | **6875** | **89.6%** | **DOMINANT.** 124 sequential corpus embeds + 1 raw query embed inside `loadCorpusEmbeddings`. This is the cold-start stall. |
| (c) buildCandidateContextPack | 0 | 0% | Pure in-memory graph walk; no network. |
| (d) chatStream TTFT | 800 | 10.4% | Live provider TLS/handshake + model warm — **DEVICE-MEASURED** via the console table (placeholder shown in the in-process script). |

**Why (b) dominates the first roundtrip:** the malicious RAW-ARGMAX pre-gate runs
`filterQuestion` BEFORE `chatStream` on every ask. On a cold launch the filter-corpus
cache is empty, so the first ask pays the full 124-sequential-embed cost serially
before the provider is even contacted — the device's ~1-min first-response stall.

### How to read the device console table (live provider-handshake span)

The in-process script can't reach a live provider. On a device/emulator:
1. In DevTools/WebView console: `localStorage.setItem('trellis_cold_start_profile','true')`, then cold-relaunch.
2. Send the first ask. The `[Trellis] cold-start first-ask phase split` table prints on first token, with the **real** `(d) chatStream TTFT` wall-clock and the now-warm `(b)` span.

## Before / After

| | Before (cold cache) | After (boot pre-warm) |
|---|---|---|
| When the 124-embed corpus cost is paid | Serially DURING the first ask, blocking the roundtrip | At boot, fire-and-forget, off the first-ask path |
| First-ask `loadCorpusEmbeddings` embeds | 124 (cold) | **0** (warm cache — proven by `cold-start-instrumentation.test.mjs`) |
| In-process `(b)` first-ask phase | 6875ms (89.6%) | ~0ms (O(1) localStorage read) once boot warm-up completes |
| Warm path (2nd+ ask) | unchanged | unchanged |

The fixed-corpus profiler still reports 6875ms because it measures the COLD-cache cost
(the work itself is unchanged); the win is *when* it is paid. The executing test is the
deterministic proof that the cost moved off the first-ask path.
