# Phase 55.1 Plan 04: ChatInput first-tap Send Summary

Send on the Ask screen now fires on the FIRST tap while the keyboard is open, by routing the
send through `onPointerDown` + `e.preventDefault()` (runs before the input blurs) into a shared
`submitMessage()` body that the Enter-key form submit also calls.

## What Was Built

**Root cause (BUGFIX-04):** The Send button was a `<button type="submit">` relying on
click/form-submit, which fires on pointer-UP ŌĆö after the focus change. On Android WebView the
first tap's blur dismisses the keyboard and the submit never reaches `handleSubmit`, forcing a
second tap.

**Fix:**
1. Extracted the send body from `handleSubmit` into a shared `submitMessage()`:
   `const submitMessage = () => { const trimmed = message.trim(); if (trimmed && !disabled) { onSend(trimmed); setMessage(''); } };`
2. `handleSubmit` now does `e.preventDefault(); submitMessage();` ŌĆö Enter-key path preserved.
3. The Send button gained `onPointerDown={(e) => { e.preventDefault(); submitMessage(); }}` so the
   tap runs BEFORE the input blurs and `preventDefault()` keeps focus (keyboard stays open). It
   also suppresses the synthetic click, so the send fires exactly once. Even if a stray click
   slipped through, `setMessage('')` already cleared the field so the trimmed guard short-circuits
   the second call (no double-send).
4. `type="submit"` and `disabled={!canSend}` kept on the button; `minWidth: 0` and the 44px button
   size untouched.
