# Phase 55.1 Plan 03: Ask-screen nav-bar keyboard-flicker fix Summary

**Keyboard-open hysteresis (open 150 / close 80) in useKeyboard via a React-free `resolveKeyboardOpen` helper, so transient mid-animation `visualViewport.resize` events no longer toggle `keyboardOpen` and reverse the BottomNavigation y-spring (BUGFIX-03).**

## Accomplishments
- Root-cause fix for the Ask-screen bottom-nav up/down oscillation: the single `MIN_KEYBOARD_HEIGHT=150` threshold was crossed in BOTH directions during the Android WebView keyboard-open animation, toggling `keyboardOpen` and reversing the nav's `y` spring.
- Added hysteresis: a delta in the `[CLOSE_KEYBOARD_HEIGHT=80, MIN_KEYBOARD_HEIGHT=150]` band keeps the prior settled state, so transient mid-animation heights cannot flip it.
- Extracted the decision into a pure, React-free `resolveKeyboardOpen` helper that is directly unit-testable under `node --test` (importing `useKeyboard.ts` fails on its `react` import).
- `getNavYTarget` mapping and the public `useKeyboard()` boolean return are unchanged; no SwipeTabContainer / root-overflow / `@capacitor/keyboard` changes.

## Files Created/Modified
- `app/src/state/keyboard-hysteresis.ts` — NEW React-free module: `MIN_KEYBOARD_HEIGHT` (open 150), `CLOSE_KEYBOARD_HEIGHT` (close 80), and the pure `resolveKeyboardOpen({ heightDelta, wasOpen, openThreshold, closeThreshold })` hysteresis decision.
- `app/src/state/useKeyboard.ts` — imports the helper + thresholds; `handleResize` now computes `nextOpen = editableFocused && resolveKeyboardOpen({ ..., wasOpen: appliedKeyboardOpen === true, ... })` instead of the bare `heightDelta > MIN_KEYBOARD_HEIGHT` toggle. Editable-focus gating, baseline tracking, width-change reset, and the boolean return are unchanged.
- `app/tests/components/BottomNavigation.slide.test.mjs` — kept all `getNavYTarget` pure-fn assertions; added source-guards (CLOSE threshold present, helper wired, no `@capacitor/keyboard`) and `resolveKeyboardOpen` contract tests (mid-range delta keeps prior state in both directions).

## Decisions Made
- Chose RESEARCH Option A (hysteresis) over Option B (debounce timer) — hysteresis is deterministic, side-effect-free, and trivially unit-testable; a timer would be timing-dependent and harder to assert under `node --test`.
- Thresholds (open 150 / close 80) are empirical and confirmed by on-device UAT (the device gate below). They are easy to tune as exported constants if UAT shows residual oscillation.
- Put the helper in its own module rather than exporting from `useKeyboard.ts`, because the test runner (`node --test`, Node 26 native TS stripping, no tsx loader) cannot resolve the `react` import that `useKeyboard.ts` pulls in.
