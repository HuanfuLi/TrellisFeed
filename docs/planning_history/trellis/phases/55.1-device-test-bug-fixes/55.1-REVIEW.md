## Summary

Reviewed the four device-test bug fixes: cross-session LLM binding (BUGFIX-01),
text-art truncation + Gemini token budget (BUGFIX-02), keyboard-nav flicker
hysteresis (BUGFIX-03), and first-tap Send (BUGFIX-04).

The BUGFIX-01 (origin-session binding), BUGFIX-03 (hysteresis), and BUGFIX-04
(pointerdown send) implementations are well-reasoned and the load-bearing
CLAUDE.md invariants (ChatInput `minWidth:0`, no `@capacitor/keyboard`, news
defer-to-streamer untouched, no explicit `blur()` on Send) are preserved.

The principal concern is in BUGFIX-02's `tightenTextArtContent` fragment-reject
gate: the new branch does **not** reject the exact dangling fragment its own
comment cites (`"Is your"`), and it actively mis-rejects valid CJK headlines —
a correctness regression for 3 of the app's 4 supported locales. The
accompanying tests are source-reading-only and never execute the helper, so
these runtime defects are unguarded. Several secondary heuristic and
robustness issues are noted below.

## Critical Issues

### CR-01: `tightenTextArtContent` fragment gate mis-handles CJK and misses its own cited example

**File:** `app/src/services/concept-feed.service.ts:788`
**Issue:**
The new reject branch is:
```ts
if (!/\s/.test(s) && s.length < 8) return null;
```
Two defects:

1. **Wrongly rejects valid CJK headlines.** Chinese and Japanese (2 of the 4
   supported locales, plus the canonical EN/ES) do not use inter-word spaces. A
   perfectly valid headline like `"世界模型为何重要"` (7 chars, no whitespace)
   satisfies `!/\s/.test(s) === true` AND `s.length < 8 === true`, so it is
   rejected and the card silently falls back to `teaser.hook || title`. After a
   `LOCALE_CHANGED` to `zh`/`ja`, short-but-correct localized text-art is
   discarded. This is a locale-correctness regression introduced by the fix, in
   a codebase that ships zh/ja bundles (CLAUDE.md i18n section).

2. **Misses the fragment it claims to catch.** The comment at lines 782-787 cites
   `"Is your"` as a target fragment. But `"Is your"` contains internal
   whitespace, so `!/\s/.test("Is your") === false` and the branch never fires —
   the dangling two-word fragment is persisted exactly as before. The gate only
   catches single-token fragments under 8 chars (e.g. `"T"`), not the
   no-terminator multi-word fragments described as the failure mode.

The unit test (`text-art-tightener.test.mjs`) only source-reads for the presence
of a `!/\s/` check + a `.length < N` compare + `return null`; it never executes
the function, so neither defect is caught.

**Fix:** Reject on a more robust signal than raw `.length`, and exempt CJK from
the whitespace heuristic. For example, gate on word-ish token count and a
codepoint length that accounts for CJK, and explicitly reject a no-terminator
dangling fragment:
```ts
if (!s) return null;
const hasCJK = /[　-鿿가-힯]/.test(s);
const looksLikeFragment =
  !hasCJK && !/\s/.test(s) && s.length < 8;          // single Latin token
const danglingNoTerminator =
  !hasCJK && /\s/.test(s) && !/[.!?…]$/.test(s) && s.length < 12; // "Is your"
if (looksLikeFragment || danglingNoTerminator) return null;
return s || null;
```
Add an executing unit test (not source-read) over representative inputs:
`"T"`→null, `"Is your"`→null, `"世界模型为何重要"`→kept, `"RAG is dead."`→kept.

## Warnings

### WR-01: `mergeTextArtContent` length heuristic discards legitimately-shorter regenerations

**File:** `app/src/services/concept-feed.service.ts:701-707`
**Issue:**
`mergeTextArtContent` treats `inc.length < exi.length` as "trade-down, keep
existing." This is a length proxy for quality that breaks on legitimate cases:
a re-style/regeneration that produces a tighter, fully-valid headline (e.g.
`"RAG is dead."` replacing a longer prior `"Retrieval augmented generation is
obsolete now"`) is rejected purely for being shorter. The defense rests on
`LOCALE_CHANGED` always stripping `textArtContent` first (so `exi` is empty at
locale-driven regen), but `_persistStylesToCache` is **also** invoked from the
normal enrichment path (`concept-feed.service.ts:678-679`) where `p.textArtContent`
can be non-empty — so a shorter-but-valid regen in that path is silently dropped.
Char-length is also a poor cross-locale comparator (zh/ja are far denser than en/es).
**Fix:** Gate on validity (tightened result is non-null after `tightenTextArtContent`)
rather than raw length, or only apply the length guard when the incoming value
fails the fragment check. At minimum, scope the `inc.length < exi.length` rule to
same-locale regenerations.

### WR-02: text-art `disableThinking` budget belt is bypassed on the streaming Gemini path

**File:** `app/src/providers/llm/index.ts:333`
**Issue:**
`geminiStream` calls `toGeminiPayload(messages)` with no `disableThinking`
argument, so `thinkingConfig.thinkingBudget: 0` is never emitted on the stream
path and `maxOutputTokens` is hard-coded to the 4096 default (the `maxTokens`
and `options` are ignored entirely in `geminiStream`). The text-art call uses
non-streaming `chatCompletion`, so this is not a live BUGFIX-02 regression today.
But the asymmetry is a latent trap: any future short-output streaming call that
relies on `disableThinking` (or a non-default `maxTokens`) will silently get
neither. **Fix:** Thread `options?.disableThinking` and `maxTokens` into
`geminiStream`'s `toGeminiPayload(...)` call for parity with `geminiCompletion`,
or document why streaming intentionally ignores both.

### WR-03: `applyKeyboardOpen` toggles the body class on every resize event even when state is unchanged

**File:** `app/src/state/useKeyboard.ts:56-61`
**Issue:**
```ts
const applyKeyboardOpen = (nextOpen: boolean) => {
  document.body.classList.toggle('keyboard-open', nextOpen);  // runs every call
  if (appliedKeyboardOpen === nextOpen) return;               // dedup is AFTER the DOM write
  ...
};
```
The early-return dedup that protects `setKeyboardOpen` runs *after* the
`classList.toggle`. Android WebView fires `visualViewport.resize` rapidly during
keyboard animation; this forces a `classList` write on every one of those events
even when the resolved state is identical. `toggle(cls, force)` with the same
force value is idempotent so there is no correctness bug, but it defeats the
intent of the dedup guard and causes unnecessary style invalidation during the
exact animation window BUGFIX-03 is trying to keep stable. **Fix:** Move the
`classList.toggle` after the dedup early-return so the body class is only written
on an actual state transition:
```ts
if (appliedKeyboardOpen === nextOpen) return;
appliedKeyboardOpen = nextOpen;
document.body.classList.toggle('keyboard-open', nextOpen);
setKeyboardOpen(nextOpen);
```

### WR-04: hysteresis close boundary is asymmetric with the open boundary (`>=` vs `>`)

**File:** `app/src/state/keyboard-hysteresis.ts:53-58`
**Issue:**
```ts
if (wasOpen) return heightDelta >= closeThreshold;   // stays open at exactly 80
return heightDelta > openThreshold;                  // stays closed at exactly 150
```
The open test is strict (`>`), the close test is inclusive (`>=`). At exactly
`heightDelta === closeThreshold` (80) the keyboard stays *open*; the band is
`[80, 150]` on close but `(150, ∞)` on open. This is internally consistent with
the tests as written, but the inclusive/exclusive split at the close edge is
undocumented and easy to regress. Because `closeThreshold` (80) is well below any
real keyboard height this is unlikely to misfire in practice, so this is a
robustness/clarity note rather than a live bug. **Fix:** Either make both
boundaries strict for symmetry, or add an inline comment explaining why the close
edge is inclusive while the open edge is exclusive.

### WR-05: cross-session UI-paint guard uses `originSessionId` capture, but the malicious-block / generic-error overlay can still flash on a switched-away session

**File:** `app/src/screens/AskScreen.tsx:357-360`
**Issue:**
The persist block correctly re-reads by `originSessionId` and only calls
`setSession`/`setStreaming(null)` when `updateUI` (origin === active) and not
aborted. However, the abort path is the *only* thing that stops a switched-away
stream from finishing its persist: `handleSelectSession`/`handleNewChat` call
`abortRef.current?.abort()` synchronously, and the post-context branch checks
`controller.signal.aborted` only at `await` boundaries (lines 279, 313). If the
post-context Q&A stream (`postContextQaService.askStreaming`) has already yielded
its final token and resolved the `for await` before the abort lands, control
proceeds past line 313's `if (controller.signal.aborted) return;` only because
abort was set in time — but if the switch happens in the microtask gap *after*
line 313 and *before* `setSession`, the `resolvePersistTarget` id-match is the
sole guard. That guard reads `sessionRef.current.id` which is correct, so the UI
is protected — but `generateSessionTitle`'s nested `setSession` (lines 371-373)
re-checks `sessionRef.current.id === titled.id` against the *titled* session, not
the active session, and fires fully async after an arbitrary LLM round-trip. A
title write can still land a `setSession` for a session the user navigated away
from if they navigate back to it later. Low-severity but worth confirming.
**Fix:** Gate the title `setSession` on `originSessionId === sessionRef.current.id`
(active session) consistent with the main persist block, not on
`titled.id === sessionRef.current.id`.

## Info

### IN-01: `mergeTextArtContent` returns the un-trimmed original on the keep paths

**File:** `app/src/services/concept-feed.service.ts:701-707`
**Issue:** The function trims into `inc`/`exi` for comparison but returns the raw
`existing`/`incoming` (untrimmed) on the keep paths. This is intentional
(preserves the persisted value verbatim) but means the trim is comparison-only —
worth a one-line comment so a future reader doesn't "fix" it into returning the
trimmed value.

### IN-02: `submitMessage` double-send guard relies on async `setMessage('')` timing

**File:** `app/src/components/ChatInput.tsx:30-36`
**Issue:** The comment (lines 25-29) claims a stray synthetic click after
pointerdown is a no-op because `setMessage('')` already cleared the field. But
`setMessage` is async (React state) — within the same event-loop turn `message`
is still the old value in the closure. The actual double-send protection is the
`onPointerDown` `e.preventDefault()` suppressing the synthetic click, not the
clear. The reasoning in the comment is misleading; if `preventDefault` ever
regresses, the trimmed-guard explanation would not save it. Tighten the comment
to credit `preventDefault` as the real guard.

### IN-03: `disableThinking` JSDoc says "No-op for non-Gemini providers" but Claude/OpenAI silently ignore it without comment at the call sites

**File:** `app/src/providers/llm/index.ts:62-69`
**Issue:** The option is documented as Gemini-only, which is accurate, but
`claudeCompletion`/`openAICompletion` accept `options` and silently drop
`disableThinking`. This is fine functionally; a brief inline note in those two
functions ("disableThinking is Gemini-only; ignored here") would prevent a future
maintainer from assuming the budget belt applies cross-provider.
