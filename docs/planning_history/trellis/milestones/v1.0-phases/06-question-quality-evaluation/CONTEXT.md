# Phase 6: Question Quality Evaluation — Initial Context

## Phase Scope
**Domain:** Intelligent filtering and routing of user questions to determine whether they should be:
1. Stored in the canonical knowledge graph (merged/refined/new node)
2. Kept only in the chat session history (not persisted to knowledge base)
3. Offered user choice to save or discard

**Current State:** The system already generates LLM-based ingestion decisions (`merge|refine|new`) with taxonomy labels, but there is **no user-facing control** over whether a question gets stored. All questions are immediately persisted.

**Goal:** Add explicit quality evaluation and user intent capture so that trivial, off-topic, or context-dependent questions don't pollute the knowledge graph.

## Locked Decisions (from discussion)

### **Quality Signals & Filtering Logic** ✓ DECIDED
**Goal:** Flag off-topic/meta-questions that shouldn't be stored in knowledge graph.

**Detection mechanism:** Hybrid pattern + LLM
- **Pattern matching first:** Detect obvious meta-questions ("What did I ask?", "Tell me a joke", "Hello", etc.)
- **LLM fallback:** For edge cases where the question is ambiguous or doesn't match patterns, ask LLM: "Is this a genuine knowledge question or off-topic chat?"

**What NOT to filter:** Allow diverse topics (Transformers → 3D printers) as long as each is a genuine educational question.

**User experience:** 
1. When a question is flagged, show a small **badge/icon** below the AI response indicating "off-topic" or "won't save"
2. User can **click the badge** to see why it was flagged and override the decision
3. If overridden, the question IS saved to knowledge graph
4. If not overridden, the question stays in session history only (not persisted to knowledge graph)

**Implementation impact:**
- Add a `flagged` field to `Question` type to track whether a question was evaluated as off-topic
- Create a filtering service with pattern rules + LLM evaluation fallback
- Modify `ChatMessage` component to show/hide the badge
- Update question persistence logic to respect the `flagged` field when deciding whether to ingest into canonical knowledge

---

### **Storage Decision Trigger for Valid Questions** ✓ DECIDED
**Approach:** **Auto-save valid questions** (current behavior)
- Questions that pass the off-topic filter are immediately saved to the knowledge graph
- No additional "Confirm?" step for every valid question (reduces friction, aligns with existing behavior)
- Users only see friction when a question is *flagged as off-topic*

---

### **User Interaction Flow for Overrides** ✓ DECIDED
**Approach:** **Simple "Yes, save anyway" button**
- When user clicks the off-topic badge, show a popover/inline message: "This looks off-topic. Save anyway?"
- Two buttons: "Yes, save anyway" | "Discard"
- No friction from checkboxes or text markers — minimal modal feel
- Aligns with "minimal UI preferred" from Phase 5

---

### **Feedback & Transparency** ✓ DECIDED
**Approach:** **Silent by default (just show badge, no explanation)**
- Show the badge/icon below the AI message
- Don't show textual explanation unless user clicks the badge
- Rationale: Keeps UI clean, respects user attention, users who want to know can click for details
- This is consistent with the "tactile, native-first" design philosophy
