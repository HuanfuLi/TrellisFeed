**Researched:** 2026-04-16
**Domain:** React 19 internationalization (react-i18next) + locale-aware LLM/TTS/YouTube providers
**Confidence:** HIGH for library versions, API shapes, and current EchoLearn file layout. MEDIUM for ZH/ES/JA TTS voice IDs (OpenAI's multilingual TTS is "optimized for English" — real-world quality varies) and for the exact TS-augmentation path that works best with flat-JSON in strict mode.

## Summary

Phase 27 introduces i18n plumbing: `react-i18next 17.0.3` + `i18next 26.0.5` + `@capacitor/device 8.0.2`, four hand-committed JSON bundles (`en/zh/es/ja`), a centralised locale injection at `app/src/providers/llm/index.ts` and `app/src/providers/tts/index.ts`, a Settings switcher at the top of `SettingsScreen`, a fourth Onboarding step, and a `--font-sans` swap via `<html data-locale>`.

The critical rule (D-07, from `feedback_i18n_translation.md`): **translation is dev-time only**. The app's LLM providers must never be reused at runtime to produce UI copy. ZH/ES/JA bundles are generated once by a Sonnet subagent, reviewed by the developer, and committed. Every new key must land in all 4 bundles in the same PR; `CLAUDE.md` (to be created at project root — does not exist yet) must enforce this with a dedicated "i18n Workflow" section.

Because the project has no backend and no HTTP-loaded language resources, the cleanest pattern is **static imports of all 4 bundles + synchronous `i18n.init()` before `createRoot`** in `main.tsx`. This gives us: no Suspense flash, `useSuspense: false`, tree-shaken-friendly JSON (each bundle ~20-40 KB gzipped), and offline-safe startup. Total added bundle size: ~50 KB gz for i18next + react-i18next + 4 locales.

**Primary recommendation:** Use `react-i18next 17 + i18next 26`, pre-bundle all 4 JSON resources, init synchronously in `main.tsx`, inject locale centrally in `chatCompletion`/`chatStream` by prepending a synthetic system message if none exists or appending a "Respond in {localeName}" suffix if one does, detect initial locale via `@capacitor/device`'s `getLanguageCode()` (native) with `navigator.language` (web) fallback and a `.split('-')[0]` normalisation to the 4 supported codes, swap `--font-sans` via `document.documentElement.setAttribute('data-locale', lng)` inside an `i18n.on('languageChanged', …)` listener, and emit a `LOCALE_CHANGED` EventBus event so `useQuestions` can abort in-flight streams.

### Pattern 1: Pre-Bundled Static Init (no Suspense flash)

**What:** All 4 JSON bundles are statically imported, so `i18n.init()` resolves synchronously and the tree is render-ready on first paint.

**When to use:** Always — EchoLearn has no backend, no lazy-loaded language resources, only 4 small bundles totalling ~120-160 KB uncompressed → ~40-50 KB gzipped across all 4.

**Why:** Avoids Suspense entirely, eliminates the "EN-flash-then-switch" failure mode cited in CONTEXT.md question H, and keeps the always-mounted 5-screen SwipeTabContainer from triggering Suspense boundaries.

```typescript
// Source: https://www.i18next.com/overview/configuration-options
// app/src/locales/index.ts
import i18n from 'i18next';
import { initReactI18next } from 'react-i18next';
import en from './en.json';
import zh from './zh.json';
import es from './es.json';
import ja from './ja.json';
import { detectInitialLocale } from '../lib/locale';
import { settingsService } from '../services/settings.service';

export const SUPPORTED_LOCALES = ['en', 'zh', 'es', 'ja'] as const;
export type SupportedLocale = typeof SUPPORTED_LOCALES[number];

// Locale NAMES for LLM system prompts (D-12) — human-readable, not BCP-47
export const LOCALE_NAMES: Record<SupportedLocale, string> = {
  en: 'English',
  zh: 'Simplified Chinese',
  es: 'Spanish',
  ja: 'Japanese',
};

const initial = detectInitialLocale(settingsService.getSync().preferences.locale);

void i18n
  .use(initReactI18next)
  .init({
    resources: {
      en: { translation: en },
      zh: { translation: zh },
      es: { translation: es },
      ja: { translation: ja },
    },
    lng: initial,
    fallbackLng: 'en',        // D-21
    interpolation: { escapeValue: false },  // React auto-escapes
    react: {
      useSuspense: false,     // resources are already loaded
    },
    saveMissing: import.meta.env.DEV,
    missingKeyHandler: import.meta.env.DEV
      ? (lngs, ns, key) => console.warn(`[i18n] missing key: ${key} (lng=${lngs.join(',')})`)
      : undefined,
  });

// Keep data-locale in sync for --font-sans swap (D-23)
i18n.on('languageChanged', (lng) => {
  document.documentElement.setAttribute('data-locale', lng);
});
document.documentElement.setAttribute('data-locale', initial);

export default i18n;
```

```typescript
// app/src/main.tsx — patched
import { StrictMode } from 'react';
import { createRoot } from 'react-dom/client';
import './index.css';
import './locales';                      // ⬅️ side-effect import runs i18n.init() before render
import App from './App.tsx';
// ...existing...
```

**Key insight:** Because `init()` returns a Promise but `resources` is preloaded, the hook layer can use `useSuspense: false` and the first `useTranslation()` call resolves synchronously. No blocking `await` needed in `main.tsx`.

### Pattern 2: Central LLM Locale Injection (D-12)

**What:** A single edit point in `app/src/providers/llm/index.ts`. Every existing call-site (`useQuestions`, `question.service`, `concept-feed` (7 call sites), `podcast`, `flashcard`, `classification`, etc.) inherits the locale directive automatically with zero per-site changes.

**Current shape** (from `app/src/providers/llm/index.ts` lines 5-8, 33-48):

```typescript
interface ChatMessage { role: 'system' | 'user' | 'assistant'; content: string; }
export interface CompletionOptions { maxTokens?: number; serviceName?: string; jsonMode?: boolean; }
export async function chatCompletion(messages: ChatMessage[], config: LLMConfig, options?: CompletionOptions): Promise<string>
export async function* chatStream(messages: ChatMessage[], config: LLMConfig, options?: CompletionOptions): AsyncGenerator<string>
```

**Recommended injection approach — pre-flight message rewrite:**

```typescript
// app/src/providers/llm/index.ts — new seam, NOT exported
import i18n from '../../locales';
import { LOCALE_NAMES, type SupportedLocale } from '../../locales';

function applyLocaleDirective(messages: ChatMessage[]): ChatMessage[] {
  const locale = (i18n.language as SupportedLocale) in LOCALE_NAMES
    ? (i18n.language as SupportedLocale)
    : 'en';
  const directive = `Respond in ${LOCALE_NAMES[locale]}.`;

  const systemIdx = messages.findIndex((m) => m.role === 'system');
  if (systemIdx === -1) {
    return [{ role: 'system', content: directive }, ...messages];
  }
  // Idempotent: don't double-inject if directive already present
  const existing = messages[systemIdx];
  if (existing.content.includes(directive)) return messages;
  const merged: ChatMessage = {
    ...existing,
    content: `${existing.content.trimEnd()}\n\n${directive}`,
  };
  return [...messages.slice(0, systemIdx), merged, ...messages.slice(systemIdx + 1)];
}

// Patch entry points — ONE line each
export async function chatCompletion(messages: ChatMessage[], config: LLMConfig, options?: CompletionOptions): Promise<string> {
  const msgs = applyLocaleDirective(messages);  // ⬅️
  switch (config.provider) { /* ...same dispatch... */ }
}
export async function* chatStream(messages: ChatMessage[], config: LLMConfig, options?: CompletionOptions): AsyncGenerator<string> {
  const msgs = applyLocaleDirective(messages);  // ⬅️
  switch (config.provider) { /* ...same dispatch... */ }
}
```

**Why this shape (vs alternatives):**
- ✅ Single edit, ~15 LoC, no per-call-site changes
- ✅ Idempotent — can't double-inject across re-entrant logic (e.g. `useQuestions` Pass 1 → Pass 2 re-prompt)
- ✅ Works with all 3 providers: OpenAI sends `messages[]` with system role; Claude extracts `role==='system'` into top-level `system` param (`providers/llm/index.ts` line 171); Gemini extracts into `systemInstruction` (line 249). All 3 respect the injected directive.
- ✅ Testable: export `applyLocaleDirective` from the module for unit tests; assert that `"Respond in Simplified Chinese."` appears in the output when `i18n.language === 'zh'`.
- ✅ English stays no-op: `"Respond in English."` is still appended, but benign. Alternative: skip injection when `locale === 'en'` to save tokens. **Recommendation: always inject** — it makes the wire-format deterministic and simplifies testing.

**Gotcha:** `web-search.service.ts` calls Tavily directly (not through `chatCompletion`). D-15 says Tavily stays EN — no change needed. ✅ Already correct.

**Gotcha:** `useQuestions.ts` line 121 has a sub-prompt with `Knowledge graph candidate context:...` followed by `WEB_SEARCH_TOOL_PROMPT`. The `applyLocaleDirective` appends a paragraph to the end, after `WEB_SEARCH_TOOL_PROMPT`. Models weight final instructions heavily, so the "respond in {locale}" directive at the end is strong. Works as designed.

### Pattern 4: Font Stack Activation (D-23)

**What:** CSS variable swap via `:root[data-locale]` selector (works with Tailwind 4 since it's pure CSS custom-property cascade — no Tailwind class needed).

```css
/* app/src/index.css — add near existing CSS vars */
:root {
  --font-sans: system-ui, -apple-system, 'Segoe UI', Roboto, 'Helvetica Neue', Arial, sans-serif;
}
:root[data-locale="zh"] {
  --font-sans: system-ui, -apple-system, 'PingFang SC', 'Hiragino Sans GB', 'Microsoft YaHei', sans-serif;
}
:root[data-locale="ja"] {
  --font-sans: system-ui, -apple-system, 'Hiragino Sans', 'Yu Gothic', Meiryo, sans-serif;
}
/* es reuses :root */

body { font-family: var(--font-sans); }
```

`data-locale` is set in two places: (1) initial in `locales/index.ts` with `document.documentElement.setAttribute('data-locale', initial)`, (2) on change via `i18n.on('languageChanged')` listener.

### Anti-Patterns to Avoid

- **Calling `llmProvider.chat()` for translation** — forbidden by D-07 and durable memory rule. No code path may go through any method that ends up hitting OpenAI/Claude/Gemini for UI copy.
- **Lazy-loading locale bundles via HTTP** — EchoLearn is local-first with no backend. All 4 JSON files must be static imports.
- **Awaiting `i18n.init()` in `main.tsx`** — unnecessary when resources are preloaded; creates a blank-screen window. The promise form is still available if future HTTP backends are added.
- **Passing locale per-call to `chatCompletion`** — violates D-12 "central injection". Don't add a `locale` option to `CompletionOptions`; read from `i18n.language` inside the provider.
- **Using `react-i18next` Suspense boundaries** — unnecessary (all resources preloaded). Setting `useSuspense: false` is correct and simpler.
- **`data-locale` on `<body>` instead of `<html>`** — CSS custom properties inherit from `:root` (which is `<html>`). Setting on `<body>` means `--font-sans` on `html` uses the default stack. Set on `<html>`.
- **Using Intl.DateTimeFormat with `i18n.language` directly without guard** — `i18n.language` is `'en'`, not `'en-US'`. `Intl.DateTimeFormat('en')` works, but for `'zh'` you want `'zh-CN'` for 24-hour + correct date order. See "Date/Intl Mapping" below.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Translation-key lookup with fallback | Your own `t(key)` wrapper | `react-i18next useTranslation` | Fallback chains, interpolation, plurals, namespaces, context — all edge cases the library handles. |
| Locale detection on native | Your own `navigator.language` parsing on Android WebView | `@capacitor/device` `getLanguageCode()` / `getLanguageTag()` | Android WebView returns inconsistent `navigator.language` (often the system WebView locale, not the app locale). Native API reads the true device setting. |
| BCP-47 parsing / matching | `split('-')[0]` handwritten | Small normalisation helper + explicit `SUPPORTED_LOCALES` allowlist | `split('-')[0]` is actually sufficient *for this scope* (en-US→en, zh-CN→zh, zh-Hans-CN→zh, ja-JP→ja), but the allowlist check after is mandatory — don't assume every split-result is a locale we ship. |
| Type generation from JSON | Hand-maintained string-literal union | `declare module 'i18next'` augmentation or a small codegen script | The augmentation is ~15 lines and automatically reflects JSON changes. See "TypeScript Key Safety Strategies". |
| Missing-key detection | Custom reporting | `missingKeyHandler` | i18next already surfaces this cleanly. Just hook it to `console.warn` in dev. |
| Structural parity check between bundles | Runtime comparison | `scripts/check-locale-parity.mjs` — read all 4 JSONs, deep-flatten to key paths, assert set equality | Fails CI if `es.json` is missing a key from `en.json`. Strict enforcement of D-03. |

**Key insight:** The one thing you DO hand-roll is a ~15-line `detectInitialLocale()` because it's tightly coupled to `settingsService.getSync().preferences.locale` persistence priority + `@capacitor/device` + `navigator.language` fallback. No library fits without awkward adapters.

## TypeScript Key Safety Strategies (D-06)

### Option A: Module augmentation (official, recommended)

**Source:** https://www.i18next.com/overview/typescript

```typescript
// app/src/locales/i18n.d.ts
import 'i18next';
import en from './en.json';

declare module 'i18next' {
  interface CustomTypeOptions {
    defaultNS: 'translation';
    resources: { translation: typeof en };
  }
}
```

**Pros:**
- 10 lines, zero build step
- Key union is auto-derived from `en.json` at typecheck time → adding a key to `en.json` + `tsc` gives immediate type error if Sonnet-generated `zh.json` missed the key
- Typos in `t('home.titlee')` fail TS compile
- Flat dot-separated keys (`t('home.title')`) work with `keySeparator: '.'` (default)

**Cons (honest):**
- JSON files can't use `as const`, so interpolation-variable inference is weaker. `t('hello', { name: 'X' })` won't complain if you forget `name`. Accept this trade-off for simplicity.
- Recursive inference of deeply-nested JSON can be slow on cold-start `tsc`. With the expected ~200-400 keys split into 10 top-level groups, this is fine. If Phase 27 bundle grows past ~1000 keys in future, revisit.

**Recommended.**

### Option B: Codegen script + string-literal union

```javascript

// Reads en.json, flattens to 'home.title' | 'home.greeting' | ... union, writes src/locales/keys.generated.ts
```

**Pros:** faster `tsc` on large bundles; explicit visible output.
**Cons:** requires new `npm run` step in `prebuild`, CI churn, manual re-run when EN changes.

**Defer.** Option A is sufficient for Phase 27 bundle size.

The official helper handles multi-namespace setups with `enableSelector: "optimize"`.

## Capacitor Device Language Detection (D-17)

**API verified from https://capacitorjs.com/docs/apis/device:**

| Method | Returns | Example | Since |
|--------|---------|---------|-------|
| `Device.getLanguageCode()` | `Promise<{ value: string }>` | `'en'` (two-char) | v1.0.0 |
| `Device.getLanguageTag()` | `Promise<{ value: string }>` | `'en-US'` (BCP 47) | v4.0.0 |

**Use `getLanguageCode()`** for Phase 27 — we only support 4 bare language codes. `getLanguageTag()` would force us to BCP-47-match and split anyway.

**Web fallback:** `@capacitor/device` works on web too, but for browsers it effectively returns `navigator.language.split('-')[0]`. Double-check by calling `Device.getLanguageCode()` on web; if it errors (unlikely), fall back to `navigator.language`.

```typescript
// app/src/lib/locale.ts — ~30 LoC
import { Capacitor } from '@capacitor/core';
import { Device } from '@capacitor/device';
import { SUPPORTED_LOCALES, type SupportedLocale } from '../locales';

export function normalizeLocale(raw: string | undefined | null): SupportedLocale {
  if (!raw) return 'en';
  const base = raw.toLowerCase().split('-')[0];
  return (SUPPORTED_LOCALES as readonly string[]).includes(base) ? (base as SupportedLocale) : 'en';
}

/**
 * Called SYNCHRONOUSLY at i18n init time.
 * Priority: (1) saved preference, (2) navigator.language (sync on web),
 * (3) default 'en'. Device.getLanguageCode() is async and is fired
 * afterwards to update if a saved pref isn't present.
 */
export function detectInitialLocale(savedPref: SupportedLocale | undefined): SupportedLocale {
  if (savedPref) return savedPref;
  if (typeof navigator !== 'undefined') {
    return normalizeLocale(navigator.language ?? navigator.languages?.[0]);
  }
  return 'en';
}

/** Native-device locale detection (async, for first-launch onboarding prefill). */
export async function detectDeviceLocale(): Promise<SupportedLocale> {
  try {
    if (Capacitor.isNativePlatform()) {
      const { value } = await Device.getLanguageCode();
      return normalizeLocale(value);
    }
  } catch { /* fall through */ }
  return normalizeLocale(typeof navigator !== 'undefined' ? navigator.language : 'en');
}
```

**Normalization heuristic verified:**
- `en-US` → `en` ✅
- `zh-CN` / `zh-Hans-CN` → `zh` ✅
- `zh-TW` → `zh` (⚠ user wants Traditional, we ship Simplified only). Acceptable for Phase 27; mitigated by Onboarding confirmation step (D-18) which lets user override.
- `es-MX` / `es-ES` → `es` ✅
- `ja-JP` → `ja` ✅
- `ko-KR` → `en` (not supported, fall back)
- `fr-FR` → `en` (not supported, fall back)

## Mid-Stream LLM Cancellation (D-22)

**Current streaming architecture** (from `providers/llm/index.ts` + `useQuestions.ts`):

- `chatStream` is an `AsyncGenerator<string>` — each provider uses `fetch` with `signal: timeoutSignal(120_000)`.
- `useQuestions.askStreaming` consumes via `for await (const token of stream)`.
- There is **no existing AbortController plumbing** that the call-site can trigger — only the timeout signal.

### Recommended cancellation pattern

**Step 1:** Thread an optional `AbortSignal` through `CompletionOptions`:

```typescript
export interface CompletionOptions {
  maxTokens?: number;
  serviceName?: string;
  jsonMode?: boolean;
  signal?: AbortSignal;          // ⬅️ new
}
```

**Step 2:** In each provider's `fetch(…, { signal })`, compose with `timeoutSignal`:

```typescript
const composed = options?.signal
  ? AbortSignal.any([options.signal, timeoutSignal(STREAM_TIMEOUT_MS)])  // Baseline 2024, Node 20+
  : timeoutSignal(STREAM_TIMEOUT_MS);
// AbortSignal.any is WebView-supported (Chromium 116+, Safari 17.4+). Capacitor 8 targets modern WebView. Verified.
```

**Step 3:** In `useQuestions.askStreaming`, subscribe to a `LOCALE_CHANGED` EventBus event and call `abortController.abort()`:

```typescript
const abortController = new AbortController();
const unsub = eventBus.subscribe('LOCALE_CHANGED', () => abortController.abort());
try {
  for await (const token of chatStream(msgs, llmConfig, { serviceName: 'ask', signal: abortController.signal })) {
    if (abortController.signal.aborted) break;
    accumulated += token;
    onToken(accumulated);
  }
} finally {
  unsub();
}
// On abort: do NOT persist the partial answer. Show a toast t('ask.localeChangedDiscarded').
```

**Step 4:** Add `LOCALE_CHANGED` to `app/src/lib/event-bus.ts`'s `AppEvent` union. Emit from the Settings switcher after `i18n.changeLanguage()` resolves.

**Known gotcha:** `AsyncGenerator` cleanup is subtle — the `for await` loop's early `break` triggers the generator's `.return()`, but the underlying `fetch` reader may still be mid-chunk. The abort signal on the fetch is the authoritative stop; the `break` is just UX.

**Minimal test:** In an integration test, fire `LOCALE_CHANGED` mid-`for await` and assert `accumulated` does not grow past the pre-event value and that `questionService.buildAndSave` is NOT called. (The current `askStreaming` always calls `buildAndSave` after the loop — we need to guard with `if (!abortController.signal.aborted)`.)

### Recommended mapping (for OpenAI TTS)

| Locale | Voice ID | Rationale |
|--------|----------|-----------|
| `en` | `'alloy'` (default) or `'nova'` | Already used. Keep user's saved preference unless overridden. |
| `zh` | `'nova'` | OpenAI voices are [all English-optimized](https://community.openai.com/t/what-languages-does-the-new-tts-api-support/485153) but "speak" the input text in its script. `nova` handles CJK pronunciations least-awkwardly in community reports. |
| `es` | `'nova'` or `'shimmer'` | Both handle Spanish reasonably. `nova` is the safest default. |
| `ja` | `'nova'` | Same reason — least-awkward CJK rendering. |

**Confidence:** MEDIUM. OpenAI's documented multilingual support is "optimized for English" — voices speak in non-English, but with English accent coloration. For Phase 27 this is acceptable (per CONTEXT.md "Claude's Discretion" — exact voices TBD). If user-reported quality is poor post-ship, a future phase can route to Azure TTS (`zh-CN-XiaoxiaoNeural`, `ja-JP-NanamiNeural`, `es-ES-ElviraNeural`) or Google Cloud TTS.

### Implementation

```typescript
// app/src/providers/tts/index.ts — synthesize() refactor
import i18n from '../../locales';
import type { SupportedLocale } from '../../locales';

const LOCALE_VOICE_FALLBACK: Record<SupportedLocale, string> = {
  en: 'alloy',
  zh: 'nova',
  es: 'nova',
  ja: 'nova',
};

export async function synthesize(text: string, config: TTSConfig): Promise<string> {
  const locale = (i18n.language as SupportedLocale) in LOCALE_VOICE_FALLBACK
    ? (i18n.language as SupportedLocale)
    : 'en';
  // If user set a voice explicitly in Settings, respect it; else locale-default
  const voice = config.voice && config.voice !== 'alloy' ? config.voice : LOCALE_VOICE_FALLBACK[locale];
  // ...rest unchanged, using `voice` instead of `config.voice`
}
```

**Tension:** CONTEXT.md D-13 says "locale → voice mapping". But the user has a voice dropdown in SettingsScreen. Options:
1. **Locale auto-selects voice, user can override in Settings** — recommended; shown above.
2. **Locale auto-selects voice, dropdown removed** — violates LLM-flexibility principle in PROJECT.md. Skip.

Planner decides. The above shape (respect user override; else locale default) is the safest default.

## Common Pitfalls

### Pitfall 1: JSON `as const` can't be applied → weaker interpolation typing
**What goes wrong:** `t('hello', { wrongKey: 'x' })` won't TypeScript-error even though `en.json` says `"hello": "Hi {{name}}"`.
**Why it happens:** JSON imports in Vite don't preserve literal types; `typeof en` is `{ hello: string }`, not `{ hello: 'Hi {{name}}' }`.
**How to avoid:** Accept this tradeoff for simplicity. The key-name itself IS type-checked (via Option A augmentation). Interpolation variable names are runtime-only. Write a small test that iterates every key and asserts the translation isn't `undefined`, catching dropped keys even if TS doesn't catch dropped interpolation vars.
**Warning signs:** Console warnings like `i18next::translator: missingKey en translation home.foo`.

### Pitfall 2: Android WebView `navigator.language` lies
**What goes wrong:** Fresh install on Android device set to Japanese still reports `navigator.language === 'en-US'`.
**Why it happens:** Android WebView's locale can diverge from the system locale, especially on older devices or custom ROMs.
**How to avoid:** On Capacitor native, prefer `@capacitor/device` `Device.getLanguageCode()`. Use `navigator.language` only as the web fallback.
**Warning signs:** UAT reports "auto-detect always shows English on Android".

### Pitfall 3: `i18n.language` vs BCP-47 for `Intl.*`
**What goes wrong:** `new Intl.DateTimeFormat('zh').format(new Date())` produces different output than `'zh-CN'` for day-of-week abbreviations.
**Why it happens:** `i18n.language` is our bare `'zh'`; Intl prefers BCP-47 tags.
**How to avoid:** Keep the `INTL_LOCALE` mapping table in `date.ts` that maps `'zh'` → `'zh-CN'`.
**Warning signs:** Visible differences between iOS/Android/web day labels on same locale.

### Pitfall 4: Tailwind 4 cascade layer order
**What goes wrong:** `:root[data-locale="zh"] { --font-sans: … }` defined in CSS doesn't override Tailwind's base layer.
**Why it happens:** Tailwind 4 uses CSS `@layer` directives; any var defined in `@layer base` can lose to a later un-layered declaration — or vice versa.
**How to avoid:** Define the `--font-sans` CSS vars in the same `@layer base` block (or un-layered — either works if consistent). Test by inspecting computed `font-family` on `<body>` after `data-locale` change.
**Warning signs:** CJK fallback fonts not rendering; still Latin-only stack active.

### Pitfall 5: Sonnet subagent over-translates proper nouns
**What goes wrong:** `"EchoLearn"` gets translated to `"回声学习"` in `zh.json`.
**Why it happens:** Sonnet is doing its job translating text; it doesn't know `EchoLearn` is the brand.
**How to avoid:** Prompt template (see "Translation Workflow Template" below) explicitly lists proper-noun DO-NOT-TRANSLATE list: `EchoLearn`, `OpenAI`, `Claude`, `Gemini`, `YouTube`, `Tavily`, `API`, `TTS`, `LLM`, `SM-2`. Also: instruct Sonnet to preserve interpolation placeholders (`{{name}}`) verbatim.
**Warning signs:** UAT reviewer spots translated brand names.

### Pitfall 6: `useTranslation` re-render storm on `changeLanguage`
**What goes wrong:** All 5 always-mounted screens + all mounted components re-render at once — UI briefly freezes on low-end Android.
**Why it happens:** `react-i18next` triggers React re-render on every `useTranslation()` subscriber when language changes. 40+ subscribers × all concurrently = frame drop.
**How to avoid:** This is actually the *intended* behaviour (D-19 "instant switch"). Mitigate by batching: wrap `i18n.changeLanguage()` + `settingsService.set()` + eventBus.emit in one event-loop turn. On very low-end devices, consider `startTransition(() => i18n.changeLanguage(lng))` (React 19 API). Profile before over-engineering.
**Warning signs:** Noticeable UI freeze on locale change on Android WebView.

### Pitfall 7: Missing keys in ZH/ES/JA after adding a new EN key
**What goes wrong:** Developer adds `t('home.newFeature')` in code; runs build; `fallbackLng: 'en'` silently shows English text on ZH/ES/JA. Ships a half-localized UI.
**Why it happens:** `fallbackLng` makes partial coverage invisible until UAT.
**How to avoid:**
1. `scripts/check-locale-parity.mjs` — read all 4 JSONs, flatten to paths, assert `en ⊆ zh, en ⊆ es, en ⊆ ja`. Fail CI.
2. CLAUDE.md section makes the rule durable.
3. `missingKeyHandler` logs to console in dev. Developer sees warnings when touching unlocalized paths.
**Warning signs:** Console warnings, CI failure, or UAT finds English strings on a non-EN locale.

```markdown
1. **Preserve existing translations** — if a key already exists in the target locale, KEEP its current value verbatim unless the EN source has changed meaning.
2. **Preserve interpolation placeholders** — `{{name}}`, `{{count}}` must appear verbatim in the output.
3. **Do NOT translate proper nouns** — keep as-is: EchoLearn, OpenAI, Claude, Gemini, YouTube, Tavily, API, TTS, LLM, SM-2, iOS, Android, Capacitor, GPT, SQLite.
4. **UI chrome tone** — concise, imperative where appropriate (button labels), friendly but not casual (onboarding copy).
5. **Length awareness** — Spanish runs ~20% longer than English; Japanese can be much shorter. Don't pad.
6. **Return only the final JSON** — no markdown fences, no commentary.

**Target locale:** {{LOCALE_NAME}} ({{LOCALE_CODE}}).
**Source (EN):** <paste `app/src/locales/en.json` here>
**Existing target:** <paste `app/src/locales/{{LOCALE_CODE}}.json` here, or "{}" if none>
```

Developer runs this as a Task-tool invocation with `subagent_type: 'general-purpose'`, copies the returned JSON back into `app/src/locales/{locale}.json`, reviews diff, commits all 4 bundles + code in one PR.

## Open Questions

1. **Should `chatCompletion` skip locale injection when `locale === 'en'`?**
   - What we know: injecting always is safe, idempotent, and keeps wire format deterministic. Costs ~5 tokens per request.
   - What's unclear: whether the token overhead matters (13 call sites × every ask × 5 tokens).
   - Recommendation: Always inject. 5 tokens/call × ~100 calls/day = 500 tokens/day ≈ $0.0005/day at Claude Sonnet rates. Negligible. Wins on simplicity + test determinism.

2. **Are OpenAI TTS voices (`nova`) acceptable for Japanese/Chinese speakers?**
   - What we know: OpenAI documents voices as "optimized for English" but community reports `nova` handles CJK text passably.
   - What's unclear: real-world user tolerance; no sample audio from EchoLearn users yet.
   - Recommendation: Ship with `nova` for ZH/JA. Add a "TTS quality" feedback toggle in SettingsScreen (future phase) if complaints arrive. Flag in UAT (D-24) for reviewer to listen to a sample per locale.

3. **Does Tailwind 4 `@layer base` require special handling for `:root[data-locale]`?**
   - What we know: Tailwind 4 uses CSS cascade layers.
   - What's unclear: whether bare `:root[data-locale="zh"]` (unlayered) overrides `@layer base { :root { --font-sans: … } }`.
   - Recommendation: Place the locale-specific overrides in the *same* `@layer base` block immediately after the default `--font-sans` declaration. Confirm during implementation by inspecting `getComputedStyle(document.body).fontFamily`. If issue arises, use `@layer base` explicitly for all locale variants.

4. **Does the Sonnet subagent need file-write permission?**
   - What we know: Task tool subagents operate in a scoped working dir.
   - What's unclear: whether the subagent writes files directly or returns the JSON in its final message.
   - Recommendation: Return JSON in the final message (simpler to review + diff). Developer copies output into `app/src/locales/{locale}.json`. Commits all 4 at once.
