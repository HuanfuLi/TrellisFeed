## Decisions Made

See frontmatter `key-decisions`. Principal call-outs:

- **Implementation-first for lib/locale.ts (Task 1 instead of Task 2 RED):** Strict TDD would have Task 2 create the failing test and then the implementation. Because `src/locales/index.ts` (Task 1) imports `detectInitialLocale` from `lib/locale.ts`, and Task 1's acceptance criteria requires `tsc -b --noEmit` exits 0 AND `npm run build` exits 0, we landed the implementation in Task 1 and used Task 2 to retrofit the TDD test coverage. Tests still validate behavior; just in reverse order from a pure RED-GREEN cadence.
- **Optional → required locale on AppPreferences (Task 1 → Task 3):** Locale was optional in Task 1 (so Task 1 compiled cleanly without Task 3's migration), then tightened to required in Task 3 once the migration path was in place. Both commits compile standalone.

**1. [Rule 3 — Blocking] Created `app/src/lib/locale.ts` in Task 1 instead of Task 2**
- **Found during:** Task 1 verification (`tsc -b --noEmit` + `npm run build`)
- **Issue:** Plan sequences lib/locale.ts in Task 2, but Task 1's `src/locales/index.ts` imports `detectInitialLocale` from it and Task 1's acceptance criteria requires the build to pass.

**2. [Rule 3 — Blocking] Added `locale?: SupportedLocale` to `AppPreferences` in Task 1 (optional), then tightened to required in Task 3**
- **Found during:** Task 1 verification (tsc resolving `settingsService.getSync().preferences.locale`)
- **Issue:** Plan's Task 1 writes `locales/index.ts` which reads `.preferences.locale`, but Task 3 is what formally adds the field to `AppPreferences`. Staging the required field in Task 1 would make Task 3 partially redundant; staging nothing leaves Task 1 uncompilable.
- **Fix:** Task 1 adds the field as optional (backward-compat for users without migration run yet); Task 3 tightens to required after the migration path is wired.
- **Files modified:** app/src/types/index.ts (Task 1: add as optional; Task 3: remove `?`)
- **Verification:** Both commits compile independently via tsc.
- **Committed in:** 507860e2 + 2b54c92b

**3. [Rule 3 — Blocking] Rewrote navigator/localStorage shims to use `Object.defineProperty`**
- **Found during:** Task 2 test run (`TypeError: Cannot set property navigator of #<Object> which has only a getter`)
- **Issue:** Node 25.9 makes `globalThis.navigator` a read-only getter, breaking the plan's inline `globalThis.navigator = { ... }` pattern.
- **Fix:** Both locale-detect and settings-locale tests use `Object.defineProperty(globalThis, 'navigator'|'localStorage', { value, configurable: true, writable: true })` and restore via `Object.defineProperty` with the original descriptor.
- **Files modified:** app/tests/lib/locale-detect.test.mjs, app/tests/services/settings-locale.test.mjs
- **Verification:** All 12 tests across both files now pass (6+6).
- **Committed in:** ac2386c1 (Task 2) + 2b54c92b (Task 3)

**4. [Rule 3 — Blocking] Added `resolveJsonModule` + `allowSyntheticDefaultImports` to `tsconfig.app.json`**
- **Found during:** Task 1 tsc run
- **Issue:** JSON imports (`import en from './en.json'`) failed under strict `verbatimModuleSyntax` without these two options.
- **Fix:** Added both to `compilerOptions` in `tsconfig.app.json`.
- **Files modified:** app/tsconfig.app.json
