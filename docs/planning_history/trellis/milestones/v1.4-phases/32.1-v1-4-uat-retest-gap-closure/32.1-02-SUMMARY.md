## Diagnosis

Both root causes confirmed by file:line citation before any code edit (Task 1, read-only).

### Cause 1 — `seenVideoIds` is per-call (Phase 31-08 regression)

**File:line:** `app/src/services/concept-feed.service.ts:685`

```ts
// Track seen YouTube videoIds to prevent duplicates within a batch (MAJOR 4 fix)
const seenVideoIds = new Set<string>();
```

The Set was declared INSIDE `generatePostBatch` (line 685, inside the function body), so every call recreates an empty Set. `refillQueue` calls `generatePostBatch` once per cycle. After the first batch, the Set is garbage-collected; the next cycle's batch starts fresh with no memory of which videoIds were already used. Cross-cycle duplicates leak.

No localStorage backing (verified — zero hits for `seenVideoIds.*localStorage`). Phase 31-08's commit message even called it "batch-level dedup" — accurate but not what users needed for cross-cycle behavior.

### Cause 2 — Cap formula saturates at low `dueConcepts.length`

**File:line:** `app/src/services/concept-feed.service.ts:906`

```ts
const dueConcepts = questions.filter(q => q.isAnchorNode);
const maxPosts = (settings.feed?.dailyGenerationCapMultiplier ?? FEED_DEFAULTS.dailyGenerationCapMultiplier) * Math.max(dueConcepts.length, 1);
if (postQueueService.getTotalGenerated() >= maxPosts) return;
```

`FEED_DEFAULTS.dailyGenerationCapMultiplier === 5` (verified at `app/src/services/settings.service.ts:5`). For a user with `dueConcepts.length === 1`, `maxPosts = 5 × max(1, 1) = 5`. The first batch generates ~3-5 posts → `getTotalGenerated()` hits 5 → `refillQueue` early-returns forever. Queue exhausts after one batch.

`getTotalGenerated()` is monotonically increasing within a day (only resets via `postQueueService.resetForNewDay()` — verified at `app/src/services/post-queue.service.ts:97-99, 113-116`).

## Fix Implementation

### A. New helper module: `app/src/services/concept-feed-dedup.ts`

Pure helper with no transitive i18n deps. Exports:

- `hasSeenVideoId(videoId: string): boolean`
- `addSeenVideoId(videoId: string): void`
- `__resetSeenVideoIdsForTesting(): void` (test-only)

Internal state: module-scope `_seenVideoIds: Set<string>` + `_trackedDay: string`. A private `maybeResetForNewDay()` runs on every call to `hasSeenVideoId` / `addSeenVideoId` and clears the Set when the day rolls over (mirrors the `today()` check in `post-queue.service.ts:36-38`). Without day-boundary reset, yesterday's videoIds would block today's generation.

### B. Wired into `concept-feed.service.ts`

- Added import (`import { hasSeenVideoId, addSeenVideoId } from './concept-feed-dedup';`) on line 15.
- Removed local `const seenVideoIds = new Set<string>();` at line 685; replaced the comment with a pointer to the helper module.
- Landscape video loop (~line 787-790): `seenVideoIds.has(v.videoId)` → `hasSeenVideoId(v.videoId)`; `seenVideoIds.add(...)` → `addSeenVideoId(...)`.
- Short video loop (~line 822-824): same substitution.
- End result: zero `seenVideoIds` symbol references in `concept-feed.service.ts`.

### C. Cap formula floor at 3

`concept-feed.service.ts:906` — replaced `Math.max(dueConcepts.length, 1)` with `Math.max(dueConcepts.length, 3)`. Inline comment documents that the floor is a *minimum*, not a ceiling — users with many due concepts still get unlimited multiplier × N.

**Why floor at 3 (not 2 or 5)?** With multiplier=5 (FEED_DEFAULTS), floor 3 gives maxPosts >= 15. Each pull-for-more serves 4 posts; a generation batch produces ~3-5 posts. 15 cap covers ~3 batches before saturation, which is enough to support meaningful exploration before any queue exhaust message fires. Floor 2 would give maxPosts=10 (only ~2 batches — still risky). Floor 5 would give maxPosts=25 — unnecessarily generous for users with few due concepts (potential token cost concern).
