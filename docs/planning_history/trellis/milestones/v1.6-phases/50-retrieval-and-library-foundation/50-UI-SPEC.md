## Typography

**4 sizes, 2 weights.** Aligned to existing screen patterns (`SavedScreen.tsx`, `LongPressMenu.tsx`, `Header.tsx`). No new font files; uses `var(--font-sans)`.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Caption | 12px | 500 (medium) | 1.4 | Result-row body snippet, filter chip label, collection post-count meta, drill-in header subtitle ("12 posts") |
| Body | 14px | 500 (medium) regular / **700 (bold)** active | 1.4 | Picker row label, search input text, search input placeholder, tab label (inactive 500 / active 700 — `SavedScreen.tsx:226` precedent), result-row title (500 → highlighted runs visually bolder via `<mark>` background, NOT via weight change), collection-list row name, sheet button labels ("Done", "Save name", "Keep collection", "Delete"), confirmation sheet body text, empty-state heading (bold 700 for salience instead of larger type — matches existing `EmptyState` at `SavedScreen.tsx:182` precedent) |
| Heading | 18px | 700 (bold) | 1.3 | Picker sheet title ("Save to..."), drill-in screen header title (rendered through existing `<Header title={...}>` which uses 1.125rem/700 — Phase 50 honors the existing Header type, no change), Rename sheet title, Delete confirmation sheet title |
| Display | (unused) | — | — | No Phase 50 surface needs a fourth size. Reserved slot; do not introduce a fourth size in this phase. |

**Why only 3 active sizes (Caption/Body/Heading) and 2 weights (500/700):**

- The phase introduces zero new typographic roles. Search-bar input reuses Body; result-row body snippet reuses Caption (smaller than title, so visual hierarchy is type-size + color, not weight).
- Highlighted matched substrings are NOT a separate weight — they use a `var(--primary-40)` background fill (see `HighlightedText` in `<Pattern 4>` of RESEARCH.md). Weight stays 500 for the title to keep the row scannable; the colored chip is the salience signal.
- 700 is used for active tab labels AND empty-state headings AND button labels AND Heading 18px. This consolidates the previously-declared 600 (semibold) into the 700 (bold) slot. The existing `<Header>` component at `app/src/components/ui/Header.tsx` already uses 700 for its title — declaring 700 (not 600) for Phase 50 Heading aligns with the live precedent.
- 500 is used for body text, caption text, inactive labels, and meta. One regular-weight value; no medium/regular split.
- Filter chip "active" state uses background-fill (see Color), not weight change. The chip label stays Caption 12/500 in both states.

**Adaptive override (NOT a fourth declared size):**

- At viewport widths **< 360px**, the 4-tab strip's per-tab label drops to **13px** via a CSS-only media query on the tab strip ONLY. This is a tactical adaptive override to prevent ellipsis on "Collections" / "コレクション" at 320px viewports; it is not a fourth declared typographic role and does not appear anywhere else in the UI. Rationale: introducing 13px as a global declared size for one edge case would expand the scale; bounding it to a `min-width` media query on a single component keeps the declared scale at 4.

**Line-height ratios:**

- Body line-height **1.4** matches `SavedRow.title` at `SavedScreen.tsx:128`. Slight tightness for compact list density. Phase 50 result rows extend the same row pattern.
- Heading line-height **1.3** matches `h1` global at `index.css:326`.
- Caption line-height **1.4** keeps snippet readable in 2-line truncation.

## Color

Theme-aware via CSS variables — light and dark theme tokens declared in `app/src/index.css:5-141` (light) and `:264-273` (dark). All Phase 50 UI uses these tokens; no new color variables introduced.

| Role | Token | Phase 50 Usage |
|------|-------|----------------|
| Dominant (60%) | `var(--surface)` / `var(--background)` | SavedScreen background, search-bar wrapper container (inherits from screen), drill-in screen background, picker sheet background (`BottomSheet.tsx:65`). |
| Secondary (30%) | `var(--surface-variant)` / `var(--card)` | Search-bar input background (matches `ChatInput` pattern at `ChatInput.tsx:95`), inactive filter chip background, post-row card background (`SavedRow.tsx:74` precedent), picker custom-collection row background on tap-down (pressed state). |
| Accent (10%) | `var(--primary-40)` | **Reserved for these specific elements only** (see list below). |
| Destructive | `var(--danger)` | **Reserved for**: delete-collection confirmation step (confirm button background), nothing else this phase. NOT used on "Remove from collection" row (that is a neutral action; uses `var(--foreground)`). |

### Accent (`var(--primary-40)`) is reserved for exactly these elements

1. Active tab underline + active tab label color (matches existing `SavedScreen.tsx:227-228`).
2. Active filter chip background fill (text color `#fff` for contrast against green; matches `ChatInput.tsx:141` Globe-toggle pattern).
3. `<mark>` highlighted matched substring background (text color `#fff`; matches RESEARCH.md Pattern 4 inline style — `background: 'var(--primary-40)', color: '#fff'`).
4. Header back-arrow color (existing `Header.tsx:62` — applies to drill-in screen too).
5. Picker sheet "Done" button background (text color `#fff`).
6. Checkbox checked state fill (custom-rendered — matches Material You spec). Unchecked checkbox border: `var(--border)`.
7. Empty-state "Clear filters" text-button color (matches existing `home.history.errorRetry` button pattern at `SavedScreen.tsx:369`).
8. Collection-row folder icon stroke when collection contains the post being saved (picker context).
9. Search-input focused border (1.5px solid; matches `ChatInput.tsx:98` recording-active border pattern).
10. Search-input magnifying-glass icon when input is focused or has text (when idle, icon color is `var(--muted-foreground)`).

**Accent is NOT used for:**

- Inactive tab labels (use `var(--muted-foreground)`).
- Inactive filter chip background (use `var(--surface-variant)`).
- Search-input border at rest (use `1.5px solid transparent` — same neutral as `ChatInput`).
- Collection-row hover/press state on Collections sub-tab (use `var(--surface-variant)` pressed background).
- "+ New collection" row label (use `var(--foreground)`; only the leading `+` icon receives accent color as a subtle CTA hint).
- Post-row hover/press state (existing `SavedRow.tsx:74` uses `var(--surface-variant)` — do not change).
- "Remove from collection" long-press menu row (use `var(--foreground)` — same as other LongPressMenu rows; not destructive in this phase since the post survives in Saved/History).

### Destructive (`var(--danger)`) is reserved for exactly these elements

1. Collection delete confirmation: confirm-button background fill (when user taps "Delete" in the drill-in kebab rename/delete sheet). Inline text "Delete \"{name}\"? Posts will remain in Saved." renders in `var(--foreground)` — only the action button itself is danger-colored.

**Destructive is NOT used for:**

- "Remove from collection" — non-destructive (post stays in Saved/History; only its collection membership removes).
- Search "Clear filters" link — neutral utility, uses `var(--primary-40)`.
- Empty-state copy — uses `var(--muted-foreground)`.

### 60/30/10 Verification on the SavedScreen with Phase 50 additions

- 60% dominant: screen background + sheet background + drill-in background.
- 30% secondary: search-bar input wrapper + result-row cards + inactive filter chips + picker custom-collection rows.
- 10% accent: active tab underline (1 line), highlighted `<mark>` spans (sub-second visible area in any given result row), Done button (44×88 in picker), focused search border (1px sliver), active filter chip (1 of 3 at most, since filters are now Concept · Source · Date and remain single-select per chip), the `+` icon on "+ New collection" (8×8).

This stays within the 10% accent budget by design — accent is salience, not surface.

## Surface-by-Surface Visual Specification

This section is prescriptive — the executor implements from here without re-asking design questions.

### Surface 1 — Search Bar (in `/saved` header area, scoped to active tab)

**Position:** Below the screen `<Header title={t('saved.title')} backTo="/home" />` and ABOVE the tab strip. Lives inside the existing scroll container (NOT inside the Header). This keeps the existing portal-vs-in-tree Header rule intact (Header is sub-screen → portals to body; search bar scrolls with content, which is fine because we WANT it to scroll off when the user scrolls the result list — feels natural on mobile, matches iOS Mail and Safari history.)

**Pinning behavior:** Per D-10, the search bar is pinned BELOW the Header — using `position: sticky; top: HEADER_HEIGHT + safe-area-top; z-index: 5` on its wrapper. The tab strip sits BELOW the search bar (also sticky, stacked). This gives: Header (portaled, anchored to viewport top) + sticky search bar (scrolls with content but sticks once it reaches the Header's bottom edge) + sticky tab strip.

```
┌─────────────────────────────────────────┐   <- Header (portaled, fixed)
│  ←   Saved                              │      height: 56px, var(--surface)
├─────────────────────────────────────────┤
│ ┌─────────────────────────────────────┐ │   <- Search bar (sticky, in-tree)
│ │ 🔍  Search...                    ✕  │ │      44px tall, var(--surface-variant)
│ └─────────────────────────────────────┘ │      var(--radius-pill), shadow-1
│  [Concept] [Source] [Date]              │   <- Filter chips (focus-only)
├─────────────────────────────────────────┤
│  Saved   Liked   History   Collections  │   <- Tab strip (sticky, in-tree)
├─────────────────────────────────────────┤
│  [post row]                             │
│  [post row]                             │
│  ...                                    │
└─────────────────────────────────────────┘
```

**Anatomy of the search bar wrapper (matches `ChatInput.tsx` family):**

| Property | Value |
|----------|-------|
| Outer wrapper | `position: sticky; top: calc(var(--safe-area-top) + 56px); z-index: 5; background: var(--surface); padding: 8px 16px 0;` |
| Inner search input | `display: flex; align-items: center; gap: 8px; padding: 0 16px; background-color: var(--surface-variant); border-radius: var(--radius-pill); height: 44px; border: 1.5px solid transparent; transition: border-color 0.2s;` — focused/has-text → `border-color: var(--primary-40)`. **Icon→input gap is 8px** (`var(--space-sm)`). |
| Search icon (left) | `lucide-react/Search`, size 18, color `var(--muted-foreground)` at rest, `var(--primary-40)` when input focused or has text. `flexShrink: 0`. |
| Input | `type="text"`, `flex: 1`, `minWidth: 0` (load-bearing per `ChatInput.tsx:169` rule), `background: transparent`, `color: var(--foreground)`, `font-size: 14px`, placeholder via `t('library.search.placeholder')`. |
| Clear-X button | `lucide-react/X`, size 18, color `var(--muted-foreground)`. Visible only when `query.length > 0`. Touch wrapper 44×44 with negative margin to maintain visual compactness. Tap → `setQuery('')` + return focus to input. `aria-label={t('library.search.clearAria')}`. |

**Focus behavior:**

- On focus, the filter chip row mounts below with a `opacity: 0 → 1` 150ms transition (no height animation — keep simple).
- On blur, IF `query` is empty AND no filter chip is active, the row unmounts with `opacity: 1 → 0` 150ms. IF query is non-empty OR any filter is active, the row stays mounted (because filters are part of the search state and dropping them silently on blur would be confusing).
- Tab switch → clears `query`, `filterConcept`, `filterSource`, `filterDate` (per RESEARCH Pitfall 8). Search bar input clears too.

### Surface 2 — Filter Chip Row (Concept · Source · Date, per D-12)

**Per CONTEXT D-12, Phase 50 ships all three filter chip types:**

- **Concept** (left) — tap opens a single-select picker sheet listing anchor titles drawn from `questionService.getAnchors()`.
- **Source** (middle) — tap opens a single-select picker sheet listing the unique set of `contextLabel` values present in the active tab's corpus (e.g., "OpenAI", "YouTube", "Tavily News").
- **Date** (right) — tap opens a single-select picker sheet listing the four presets: Today / Last 7 days / Last 30 days / All time.

All three chips appear inline below the search bar on search-bar focus, and collapse on blur (per D-11). Active filter chips render in the accent-fill style and show an "X" affordance to clear; inactive chips render with the chip's label only.

**Visual contract:**

| Property | Value |
|----------|-------|
| Row container | `display: flex; gap: 8px; padding: 8px 16px 0; overflow-x: auto; -webkit-overflow-scrolling: touch; scrollbar-width: none;` (hide scrollbar on iOS). 2. **Body snippet line BELOW the existing contextLabel meta:** A 3rd text line, 120 characters centered on the first body match, with highlighted matched runs. If snippet starts mid-word or ends mid-word, prepend/append `…` (Unicode ellipsis). NOT shown if no body match (only title matched) — only the highlighted title appears in that case.

Snippet extraction algorithm (RESEARCH Pattern 4 — already specified, repeated here for completeness):

```typescript
function extractSnippet(body: string, firstMatchStart: number, maxChars = 120): { text: string; offset: number } {
  const half = Math.floor(maxChars / 2);  // 60
  const rawStart = Math.max(0, firstMatchStart - half);
  const rawEnd = Math.min(body.length, rawStart + maxChars);
  const text = body.slice(rawStart, rawEnd);
  const prefix = rawStart > 0 ? '…' : '';
  const suffix = rawEnd < body.length ? '…' : '';
  return { text: prefix + text + suffix, offset: rawStart };
}
```

Indices passed to `<HighlightedText>` for the snippet MUST be re-based against `offset` (subtract `offset` from each `[start, end]` pair, clamp to `[0, text.length]`, drop pairs entirely outside the snippet window).
