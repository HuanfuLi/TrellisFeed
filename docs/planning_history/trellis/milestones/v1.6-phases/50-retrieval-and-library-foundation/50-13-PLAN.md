<objective>
Close gap G5 from 50-UAT.md Test 5 clarification.

Operator reported (2026-05-18): FilterPickerSheet drawer is "not optimized for scrolling — no safe area above/below viewable area, user sees flicker or end of page when scrolled too fast at boundary". Screenshot context: FilterPickerSheet open from /saved with 'System' query, list showing Few-Shot Learning / Transformer Architecture / Prompt Engineering / SSDs vs RAM / Feynman Technique. Operator's prescription: "rigid boundary instead of flexible".

Root cause: the BottomSheet's inner scroll container (the 20px-rounded drawer body — `BottomSheet.tsx:51-104`, the `<div onClick={stop} style={{ ... overflowY: 'auto', borderRadius: '20px 20px 0 0' ... }}>`) uses default `overscroll-behavior: auto`. On Android WebView and iOS WKWebView, fast-scroll past the boundary triggers a rubberband / scroll-chaining effect that briefly exposes content beneath the drawer's rounded mask. With `overscroll-behavior: contain` declared on THAT element (the one that owns both the scroll AND the rounded mask), the scroll cannot chain to ancestors and cannot rubberband past the drawer's bounds.

**Fix scope:** apply at BottomSheet level (revised from the original FilterPickerSheet-only scope per checker BLOCKER 1). The original Option A — wrapping FilterPickerSheet's children in a nested div with `overscrollBehavior: 'contain'` — was a no-op because that wrapper has no `overflow: auto` rule and is therefore NOT a scroll container. The CSS `overscroll-behavior` property only applies to scrolling elements. The right element to touch is BottomSheet's inner sheet `<div>` at lines 51-104 — the one that owns `overflowY: 'auto'` (line 74) AND the `borderRadius: '20px 20px 0 0'` rounded mask (line 66). Putting the property on the SAME element keeps the rigid boundary aligned with the visual clip — a single source of truth that all BottomSheet consumers (CollectionPickerSheet, FilterPickerSheet, Rename/Delete sheets in SavedScreen, LongPressMenu) inherit. The behavior change is conservative — `overscroll-behavior: contain` only changes behavior at scroll boundaries; idle scrolling is identical. Phase 32.1 establishes precedent for treating drawer scroll boundaries as load-bearing on Android WebView.

Purpose: Make every BottomSheet feel like a rigid panel — fast-scrolling near top/bottom hits a hard stop, no flicker, no end-of-page bleed.
Output: Updated BottomSheet.tsx with explicit overscroll-behavior styling on the inner scroll container + a source-reading regression test that asserts the style key lives on the SAME element as `overflowY: 'auto'` + a manual-only verification row in 50-VALIDATION.md so the on-device behavior is re-tested by the operator.
</objective>
