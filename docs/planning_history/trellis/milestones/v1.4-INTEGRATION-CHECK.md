## Seam-by-Seam Findings

### Seam 1 (WARN) — Phase 29 TD-01 regressed by Phase 30/31 refactor

**Decision IDs affected:** 29-TD-01 (curiosity signal wiring)

**Evidence:**
- `app/src/services/plannerAutoGen.service.ts:115-116` still wires `checkInSignals` → `computeHints` → WIRED.
- `app/src/services/concept-feed.service.ts` — `applyStrategyBias` function is **gone**. `computeHints` no longer imported.
  - Grep `computeHints` returns: 2 hits in `orchestration-strategy.service.ts` (definition) + 1 hit in `plannerAutoGen.service.ts:116`. Zero hits in concept-feed.
  - `concept-feed.service.ts:251` still calls `plannerService.getRecentSignals()` but only to populate `plannerSignals` for LLM prompt context — the **strategy-bias sort** on `priorityConceptIds` is gone.
- **Test evidence:**
  - `tests/services/concept-feed-strategy.test.mjs`: 11 tests, **6 pass / 5 fail**. The 5 failures all assert structural presence of `applyStrategyBias` and its internal `computeHints` call.
  - `tests/services/orchestration-strategy.test.mjs`: 10 tests, **9 pass / 1 fail**. The one failure: `TD-01 plumbing: concept-feed.service.ts applyStrategyBias passes checkInSignals to computeHints`.

**Impact:** Phase 29 VERIFICATION.md's observable truth #1 ("TD-01: checkInSignals threaded to computeHints at both call sites") is NO LONGER TRUE. The concept-feed branch of TD-01 was silently dropped when Phase 30/31 rewrote the daily-post pipeline around `postQueueService` + `buildConceptBatch`. Strategy hints no longer re-order posts by user curiosity/confusion signals in the feed output.

**Not a blocker because:** `plannerAutoGen.service.ts` still receives the strategy bias (so planner-generated suggestions are still signal-aware). The feed itself no longer biases post order to recent signals — but posts are now generated per-concept from SM-2 due list, so signal-based re-ordering is arguably obsolete by design. However, the regression is silent: 5 tests assert the old wiring and fail, and 29-VERIFICATION.md is now stale.

**Severity:** WARN — tests fail and 29-VERIFICATION.md is stale; functionality is subsumed by Phase 31's generation-time concept prioritization (`buildConceptBatch` D-14: weak concepts get 2 posts), so no user-visible regression.

### Seam 10 (WARN) — Phase 31 LeafState type mismatch in buildConceptBatch

**Decision IDs affected:** 31-D-14 (important concept gets 2 posts)

**Evidence:**

Users with `yellow`-only stale anchors may see 1 post per concept instead of 2.

**Severity:** WARN — type-level bug and `npx tsc -b --noEmit` flags it. Behaviorally minor (most stale concepts hit `easeFactor < 1.5` branch first). Fix: replace `'dying'` with `'yellow'` or remove the dead comparison.

## Flow Walkthroughs (E2E)

### Flow 1 — First-time user fresh install (D-43)

1. User opens app for the first time, completes onboarding (`/onboarding`).
2. Redirect to `/home`. `questions.length === 0`.
3. HomeScreen calls `conceptFeedService.getDailyPosts(questions)` → `concept-feed.service.ts:1112` returns `STARTER_POSTS`.
4. `quotaAnchorIds` is empty (no anchors yet) → `conceptQuota === 0` → VineProgress NOT rendered (HomeScreen:615).
5. Empty-state guard: `conceptQuota === 0 && dailyPosts.length > 0 && questions.length > 0` — fails because `questions.length === 0` → empty state NOT rendered.
6. InlineInfoFlow renders the 3 STARTER_POSTS in tutorial order.

**Status:** COMPLETE. (Post-31-10 fix. Previously UAT-14 failed because the `questions.length > 0` guard was missing.)

### Flow 2 — User reads a post and marks concept explored (D-04..D-06)

1. User taps a concept post on HomeScreen → `navigate('/posts/${postId}', { state: { post } })`.
2. PostDetailScreen mounts. `resolvedAnchorId` effect resolves anchor (`:103-108`, uses `getAnchorIdForPost`).
3. Detector A attaches IntersectionObserver on 70% sentinel (`:124-137`).
4. Detector B starts 30s timer (`:139-149`).
5. Essay stream starts via AbortController-scoped useEffect (`:234`).
6. On ANY detector firing (scroll 70%, 30s dwell, or follow-up via Detector C at `:409`), `emitExplored(resolvedAnchorId)`:
   - `dailyReadService.markExplored(anchorId)` → localStorage update.
   - `eventBus.emit({ type: 'CONCEPT_EXPLORED', payload: { anchorId } })`.
7. HomeScreen's subscriber fires → `setExploredAnchors(dailyReadService.getExploredAnchors())` → `exploredCount` recomputes → VineProgress `concepts` prop updates → checklist entry flips to `explored: true`.
8. If all concepts explored, HomeScreen:459-469 awards +1 trellis credit, fires confetti, shows toast (`creditToast`).

**Status:** COMPLETE.

### Flow 3 — Locale switch aborts essay but preserves reading detection

1. User on PostDetailScreen, essay streaming.
2. User switches locale in Settings.
3. `LOCALE_CHANGED` event fires. PostDetailScreen:241-243 subscriber calls `abortController.abort(new DOMException('Locale changed', 'AbortError'))`.
4. Essay stream halts; catch block at `:325-327` checks `abortController.signal.aborted` → silent cancel (no error toast).
5. Detectors A, B, C are in separate useEffects that do NOT reference abortController → they keep running.
6. If user stays on post 30s, Detector B still fires → CONCEPT_EXPLORED still emitted.

**Status:** COMPLETE. Clean separation.

### Flow 4 — Pull-up load more (D-10, D-11, post 31-08 fix)

1. User scrolls to bottom, pulls up past threshold.
2. HomeScreen calls `loadMore()` → `conceptFeedService.generateMorePosts(questions)`.
3. If queue has ≥4 posts: dequeue 4, return immediately.
4. If queue empty: `await refillQueue(questions)` (Phase 31-08 fix: changed from fire-and-forget) → retry dequeue.
5. `buildConceptBatch` (`:729-752`) filters out concepts already pending in queue (`pendingIds`), but NO LONGER filters `exploredIds` (31-08 fix) → concepts can cycle indefinitely until daily generation cap.
6. `generatePostBatch` (`:759`) deduplicates YouTube videoIds via `seenVideoIds` Set (31-08 fix).
7. 4 new posts appended to InlineInfoFlow.

**Status:** COMPLETE post-31-08. (Previously UAT-7 + UAT-13 reported as blockers.)

### Flow 5 — SuggestionCard topic tap (D-24, D-25)

1. SuggestionCard renders with 4 LLM-generated topics (31-10 fix).
2. User taps a topic button → `onTap(topic)` → `navigate('/ask', { state: { autoSend: topic } })`.
3. AskScreen's `useEffect` on location (`:382-395`) reads `state.autoSend`.
4. Guards: `lastAutoPrompt` ref prevents re-fire on re-render.
5. Opens fresh session via `startNewSession(sessionRef.current)` + `setSession(fresh)`.
6. Calls `void handleSend(prompt)` → full QA pipeline (off-topic filter, mindmap storage, SM-2).
7. Clears nav state via `window.history.replaceState({}, '')` → back-nav doesn't re-send.

**Status:** COMPLETE.

### Flow 6 — Inline video playback + swipe-away (D-28, D-29, post 31-09 fix)

1. User taps video post play icon → `setVideoPlaying(post.id)` on InlineInfoFlow's parent state.
2. Iframe renders with `autoplay=1`. Transparent overlay (`:294-298` landscape, `:393-397` short) prevents YouTube iframe from capturing scroll touches.
3. Close button (X) at top-right stops playback: `setVideoPlaying(null)`.
4. If user taps another video: unified state means the first stops (only one `videoPlaying` at a time).
5. If user swipes to another tab:
   - `SwipeTabContext.swipeProgress` value change detected at InfoFlow:936-938.
   - `if (Math.round(v) !== 0) setVideoPlaying(null)` → video stops.
6. If browser tab hidden (`document.hidden`): `setVideoPlaying(null)` at `:928-930`.

**Status:** COMPLETE post-31-09. (Previously UAT-4 + UAT-5 reported as blocker/major.)
