# Phase 55.1: Device-Test Bug Fixes - Research

**Researched:** 2026-05-21
**Domain:** React 19 + framer-motion streaming/state bugs, Android WebView keyboard/layout, multi-provider LLM token budgeting
**Confidence:** HIGH (all four root causes traced to specific file:line in the actual codebase; BUGFIX-02 also externally corroborated)

## Summary

All four bugs are root-caused in this document with file:line evidence. Three of the four
are **logic/state bugs with deterministic root causes** (BUGFIX-01, BUGFIX-02, BUGFIX-04);
one is an Android-WebView keyboard-animation race (BUGFIX-03). None require new libraries —
every fix is a surgical change to existing files, and several of those files carry
load-bearing CLAUDE.md invariants the fix must NOT regress.

- **BUGFIX-01 (session leakage):** `AskScreen.generateAiReply` writes the streamed answer to
  `sessionRef.current` (the *currently active* session) at persist time, not the session the
  request originated in. `handleNewChat` / `handleSelectSession` do **not** abort the in-flight
  `abortRef` controller, so an A-session stream finishes and saves into session B. The
  `generatingRef` guard also silently drops the new session's send. Fix = capture the
  originating session id, abort on session switch, and gate every persist on id-match.
- **BUGFIX-02 (text-art truncation):** `generateTextArtContent` calls `chatCompletion(..., { maxTokens: 80 })`
  (`concept-feed.service.ts:793`). On Gemini 2.5/3 "thinking" models, thinking tokens are
  counted against `maxOutputTokens`, so an 80-token budget is consumed by reasoning and the
  visible text comes back empty or a single fragment ("T", "Is your") with `finishReason:
  MAX_TOKENS`. The fragment is then persisted via `_persistStylesToCache`, permanently
  corrupting the post. The locale switch is the *trigger* (the `LOCALE_CHANGED` subscriber
  strips `textArtContent` → background regen runs under the now-Gemini provider).
- **BUGFIX-03 (nav flicker):** `BottomNavigation` animates `y → '100%'` whenever
  `useKeyboard()` reports `keyboardOpen=true`. The Android WebView fires many
  `visualViewport.resize` events during the keyboard animation, and `useKeyboard`'s threshold
  crossing (`MIN_KEYBOARD_HEIGHT=150`) can toggle the boolean false→true→false mid-animation,
  reversing the spring = visible up/down flicker.
- **BUGFIX-04 (first Send tap dismissed):** The Send button is `<button type="submit">` with the
  default `onClick`/form-submit path. On Android WebView, tapping outside the focused input
  fires `focusout`/blur first; the blur is consumed and the click does not reach `handleSubmit`
  on the first tap. Fix = fire send on `pointerdown` with `preventDefault()` to preserve focus.

**Primary recommendation:** Treat 01 and 02 as correctness bugs with runtime regression tests;
treat 03 and 04 as device-only UI bugs guarded primarily by source-reading negative-assertion
tests (matching the existing `SwipeTabContainer.resize-guard` / `ChatInput.flex-shrink` pattern)
plus mandatory in-browser/on-device UAT. The Node suite gives false-green on keyboard/platform work.

## BUGFIX-01 — Cross-session LLM response leakage

### Confirmed root cause

Three compounding defects in `app/src/screens/AskScreen.tsx`:

1. **Persist-to-active-not-originating** (`AskScreen.tsx:329-332`):
   ```ts
   const latest = sessionRef.current;                                  // ← the CURRENTLY active session
   const updated: ChatSession = { ...latest, messages: [...latest.messages, aiMsg] };
   sessionService.save(updated);
   ```
   `generateAiReply` reads `sessionRef.current` *after* all the streaming awaits. If the user
   switched to a different session during the stream, `sessionRef.current` is now the new
   session, so the answer is appended to and saved under the wrong session. This is the exact
   "3rd-session answer under 5th-session question" symptom.

2. **No abort on session switch.** `handleNewChat` (`AskScreen.tsx:461-466`) and
   `handleSelectSession` (`AskScreen.tsx:468-479`) call `setSession(...)` + `setStreaming(null)`
   but **never call `abortRef.current?.abort()`**. The in-flight stream from the previous session
   keeps running and ultimately hits the persist block above. (The only abort sites today are the
   unmount effect at `:208` and the start of the *next* `generateAiReply` at `:255`.)

3. **`generatingRef` silently drops the new send** (`AskScreen.tsx:250-251`):
   ```ts
   if (generatingRef.current) return;   // drops the new session's send while A is still streaming
   generatingRef.current = true;
   ```
   During rapid ask→new-session→ask, the second `ask` can be dropped because the first stream
   is still marked in-flight — so the new session shows nothing while the old stream's result
   lands in it.

The streaming overlay state (`setStreaming` at `:260, :272, :293`) is also a single
screen-level value with no session binding, so transient streamed tokens render under whichever
session is currently displayed.

### Recommended fix approach

In `AskScreen.tsx`:

1. **Capture the originating session id** at the top of `generateAiReply` (before the first await):
   `const originSessionId = sessionRef.current.id;`
2. **Gate every state write and persist on id-match.** Replace `const latest = sessionRef.current`
   (line 329) with a re-read by id: `const latest = sessionService.getById(originSessionId);`
   and bail (still persisting the answer to the *correct* session) if `sessionRef.current.id !== originSessionId`
   for the *UI* updates (`setSession`/`setStreaming`) — i.e. persist to `originSessionId` always,
   update React state only when the origin is still the displayed session.
3. **Abort on session switch.** Add `abortRef.current?.abort()` to `handleNewChat`,
   `handleSelectSession`, and `handleDeleteSession` (when deleting the active session) before
   `setSession`. Make `generateAiReply` honor the abort in its persist block (it already checks
   `controller.signal.aborted` at `:298, :335`).
4. **Reconsider the `generatingRef` drop.** Either (a) abort the prior stream and start the new
   one, or (b) keep the guard but ensure the dropped call doesn't leave the new session in a
   permanent "streaming" placeholder. Preferred: abort-and-restart so each session's send is honored.

This is a per-request session-binding fix; it does NOT touch `useQuestions.askStreaming`'s message
array (Phase 35 byte-stability) — the binding lives entirely in the AskScreen caller.

### CLAUDE.md invariants the fix must preserve

- **Ask-chat system prompt byte-stability** — do not restructure `useQuestions.askStreaming`'s
  message array; the fix is in AskScreen only. (`tests/state/useQuestions-system-prompt-stability.test.mjs`)
- **No-refresh assumption** — the fix relies on re-reading `sessionService.getById(originSessionId)`
  rather than a stale React snapshot, consistent with the always-mounted re-read discipline.
- **LOCALE_CHANGED abort plumbing** in `useQuestions` (the shared `AbortController` at
  `useQuestions.ts:138`) is separate from AskScreen's `abortRef`; keep both. Do not collapse them.

### Regression-test strategy

- **Runtime test (preferred).** New file `tests/screens/AskScreen.session-binding.test.mjs`.
  Extract the persist-decision into a pure helper (mirrors the existing `renderRecentQuestionsMarker`
  extraction pattern at `AskScreen.tsx:52`) — e.g. `resolvePersistTarget({ originSessionId, activeSessionId })`
  returning which session id to write and whether to update UI. Contract-test: origin≠active →
  writes to origin, no UI update; origin===active → writes + UI update. This avoids the
  framer-motion/react-router .tsx import chain (see `BottomNavigation.slide.test.mjs` note).
- **Source-reading guard (belt).** Assert `handleNewChat` / `handleSelectSession` contain
  `abortRef.current?.abort()` and that `generateAiReply` captures an `originSessionId` before
  the first await. Pattern: `SwipeTabContainer.resize-guard.test.mjs`.

### Risks / unknowns

- The bug reproduces only under *rapid* interaction (stream still in flight when the user switches).
  Plan must include an on-device UAT script: ask → immediately New Chat → ask → New Chat, repeated,
  and confirm no answer crosses sessions. Node test covers the binding logic, not the timing.
- `processSessionIfNeeded` / `generateSessionPostsIfNeeded` fire on session switch and read
  `sessionService.getById(latest.id)` — verify the abort doesn't strand a half-streamed message
  that then gets flashcard-processed. Plan should confirm the aborted placeholder is discarded.

## BUGFIX-02 — Text-art posts truncate after provider/locale switch

### Confirmed root cause

`app/src/services/concept-feed.service.ts:790-794`:
```ts
const result = await chatCompletion(
  [{ role: 'user', content: prompt }],
  settings.llm,
  { maxTokens: 80, serviceName: 'text-art' },   // ← 80-token output budget
);
```
This flows to `geminiCompletion` (`providers/llm/index.ts:291`) → `toGeminiPayload` (`:273`)
which sets `generationConfig.maxOutputTokens: 80` (`:285`). **On Gemini 2.5/3 "thinking" models,
thinking tokens are counted against `maxOutputTokens`.** With only 80 tokens, the model's internal
reasoning consumes the budget and the visible text comes back empty or a tiny fragment, with
`finishReason: MAX_TOKENS`. The code reads whatever partial text exists:
```ts
return candidates?.[0]?.content?.parts?.[0]?.text ?? '';   // index.ts:309 — '' or "T" / "Is your"
```
`tightenTextArtContent` (`:728`) keeps the fragment as-is (it only trims/shortens, never rejects a
too-short result), and `_persistStylesToCache` (`:684-693`) writes the truncated fragment back into
the cached post — **permanently corrupting `textArtContent`** for that post.

**The trigger chain:**
1. User switches LLM provider to Gemini (a thinking model).
2. User (or the app) switches locale → the `LOCALE_CHANGED` subscriber (`:700-710`) strips
   `textArtContent` from every cached text-art post so it will regenerate under the new locale.
3. `_backgroundGenerateTextArt` (`:673`) sees `!p.textArtContent` and regenerates via the now-Gemini
   provider with `maxTokens: 80` → truncated content persisted → cards collapse to "T"/"Is your".

There is **no provider-change handler** that strips content, but any regeneration path
(`generateTextArtContent` / `_backgroundGenerateTextArt`) after a provider switch hits the same
80-token wall. The rendering code (`InfoFlow.tsx:482`, `PostDetailScreen.tsx:986`) reads the
persisted field faithfully — the corruption is upstream at generation/persist time.

**External corroboration (HIGH):** Multiple Gemini API issues confirm thinking tokens consume
`maxOutputTokens`, producing empty/truncated text with `finishReason: MAX_TOKENS` at low budgets
(e.g. 35–80). `[CITED: github.com/googleapis/python-genai/issues/782, /issues/2062;
discuss.ai.google.dev/t/finishreason-max-tokens-but-text-is-empty/81874]`

### Recommended fix approach

Two independent fixes; do BOTH:

1. **Raise the text-art token budget and/or disable Gemini thinking.**
   - Bump `maxTokens` from `80` to a value that survives thinking overhead (e.g. `512`). The
     output is still constrained to one sentence by `tightenTextArtContent`, so a larger budget
     does not lengthen the rendered card — it just stops the thinking budget from starving the text.
     `[ASSUMED — exact safe value needs confirmation; 512 is a conservative estimate, not measured]`
   - **Better:** add `thinkingConfig: { thinkingBudget: 0 }` to `toGeminiPayload`'s
     `generationConfig` (`index.ts:284`) to disable thinking for short non-reasoning calls. This is
     the Google-recommended fix for thinking models on low-token calls.
     `[CITED: discuss.ai.google.dev — disable thinking via thinkingConfig]` `[ASSUMED — field name/shape must be verified against current Gemini v1beta API before locking]`
2. **Treat persisted post content as immutable + reject bad regenerations.**
   - In `tightenTextArtContent` (`:728`) or at the persist site, **reject** results that are empty
     or below a sane minimum length (e.g. < 8 chars / no word boundary) so a truncated fragment is
     never persisted — fall back to `teaser.hook || title` (the existing fallback at `:816`) instead.
   - Guard `_persistStylesToCache` (`:691`) so it **never overwrites a non-empty `textArtContent`
     with a shorter one** — `info.textArtContent ?? p.textArtContent` currently lets a regenerated
     fragment replace good content. Change to keep the existing content unless the new one passes
     the minimum-length gate.
   - Reconsider the `LOCALE_CHANGED` strip (`:700-710`): stripping on every locale change is what
     exposes existing good posts to a bad regeneration. Options: (a) only strip if the new provider
     is known-safe, or (b) keep the strip but rely on fix #1 + the reject gate so regeneration
     can't corrupt. Decision needed during planning.

### CLAUDE.md invariants the fix must preserve

- **News pipeline empty-bodyMarkdown** — news posts intentionally set `bodyMarkdown: ''`. The
  reject-empty gate must apply to `textArtContent` only, NOT to `bodyMarkdown` (different field,
  different lifecycle). Do not "fix" news posts into eager generation.
- **New-day rehydration / cache date logic** (`loadCache` returns null on stale date) — the fix
  must not alter the midnight stale-rejection behavior. Tests: `concept-feed-cache-date.test.mjs`.
- **One signal per semantic event** — do not add a new PROVIDER_CHANGED event if a guard at the
  persist site suffices.

### Regression-test strategy

- **Runtime test (preferred).** Extend `tests/services/text-art-tightener.test.mjs` and/or add
  `tests/services/text-art-no-truncate-on-regen.test.mjs`:
  - Assert `tightenTextArtContent` returns `null` (rejects) for empty / single-fragment input
    like `"T"` and `"Is your"` (so the fallback fires instead of persisting the fragment).
  - Assert a `_persistStylesToCache`-style merge keeps existing non-empty content when the new
    value is shorter/empty. Extract the merge into a pure helper to test without the LLM.
  - Assert the LOCALE_CHANGED handler does not corrupt a post whose regeneration fails.
- **Provider-payload guard.** Source-read `providers/llm/index.ts` to assert the Gemini path
  either raises the budget or sets `thinkingConfig`/`thinkingBudget` for short calls — lock the fix
  so a future refactor doesn't drop back to bare `maxOutputTokens: 80`.

### Risks / unknowns

- The exact safe `maxTokens` value and the precise `thinkingConfig` field shape for the current
  Gemini `v1beta` `generateContent` API must be verified before locking (tagged `[ASSUMED]`).
  Planning should include a quick API-doc check / device test against the user's actual Gemini model.
- Other low-`maxTokens` call sites may have the same latent bug under Gemini thinking models.
  Grep for `maxTokens:` with small values (e.g. session-title at AskScreen.tsx uses `serviceName: 'title'`
  with default budget — verify). Plan should audit all small-budget calls, not just text-art.
- Already-corrupted posts in a user's cache won't self-heal unless regenerated. Plan may need a
  one-time "re-strip + regenerate under fixed budget" path, or accept that corrupted posts clear
  on next daily rebuild.

## BUGFIX-03 — Ask-screen nav-bar flicker on keyboard open

### Confirmed root cause

The bottom nav is **designed** to slide off-screen when the keyboard opens:
`BottomNavigation.tsx:174` → `animate={{ y: getNavYTarget(isTopLevelScreen, keyboardOpen) }}`
with `getNavYTarget` returning `'100%'` when `keyboardOpen` is true (`:46-48`), animated by
`SLIDE_SPRING` (`:40`).

The flicker comes from `keyboardOpen` **toggling during the keyboard-open animation**.
`useKeyboard.ts:handleResize` (`:59-84`) recomputes open/closed on every `visualViewport.resize`,
`window.resize`, `focusin`, and `focusout` event. On Android WebView the keyboard opens over
several frames, firing many resize events whose `heightDelta` crosses the `MIN_KEYBOARD_HEIGHT=150`
threshold (`:3, :82`) in both directions during the transition. Each crossing flips `keyboardOpen`,
which reverses the nav's `y` spring → the nav animates up and down = the observed flicker.

Contributing factors:
- The nav is `position: fixed; bottom: 0` (`BottomNavigation.tsx:177-178`). Under `adjustResize`
  the WebView already lifts `bottom: 0` above the keyboard, so the *additional* React `y: '100%'`
  slide is layered on top of a viewport that is itself resizing — two animation owners on one element.
- The threshold-crossing has no hysteresis or debounce, so transient mid-animation heights toggle
  the state. (Compare: `SwipeTabContainer.resync` deliberately ignores height-only events — the
  nav's keyboard handling has no equivalent guard.)

### Recommended fix approach

Pick one (planning to decide; option A is lowest-risk):

- **Option A — debounce / hysteresis in `useKeyboard`.** Add a small settle delay (e.g. ignore
  state flips within ~120ms of the last flip) or hysteresis (separate open/close thresholds, e.g.
  open at `heightDelta > 150`, close at `heightDelta < 80`) so transient mid-animation crossings
  don't toggle `keyboardOpen`. This is the smallest change and fixes the flicker at the source.
- **Option B — make the nav's slide a CSS transition that ignores intermediate state.** Keep
  `keyboardOpen` but only animate on a *settled* value (debounced), or drop the slide-down entirely
  and let `position: fixed; bottom: 0` + `adjustResize` lift the nav naturally (verify the nav doesn't
  overlap the input island first). Higher risk — touches the D-06 slide design.
- **Option C — disable the spring while keyboard transitions.** Set `transition` to a faster/no
  spring when entering keyboard state so any reversal is imperceptible. Cosmetic, doesn't fix the toggle.

Recommended: **Option A** (hysteresis/debounce in `useKeyboard`), because it fixes the root toggle
without touching the load-bearing SwipeTabContainer/overflow machinery or the D-06 nav-slide design.

### CLAUDE.md invariants the fix must preserve (CRITICAL)

- **Do NOT install `@capacitor/keyboard` with `resize:'none'`** as a workaround. Fix at the
  React/state level.
- **SwipeTabContainer resize + keyboard:** do not remove the `newWidth === screenWidthRef.current`
  early-return in `resync()`, nor the `onFocusOut` re-snap / `scrollLeft = 0` reset.
  (`tests/components/SwipeTabContainer.resize-guard.test.mjs`)
- **Root overflow clip — both axes:** keep `html, body { overflow: hidden }`, App-root
  `overflowX: 'hidden'`, and the `scrollLeft = 0` reset. (`tests/layout/root-horizontal-clip.test.mjs`)
- **Header positioning:** do not add `transform`/`will-change`/`filter`/`contain`/`perspective` to
  any nav/header ancestor.
- **BottomNavigation slide contract:** `getNavYTarget` semantics (`tests/components/BottomNavigation.slide.test.mjs`)
  must still hold (`0` only when top-level AND keyboard closed). A debounce changes *when*
  `keyboardOpen` flips, not the mapping — the existing test stays valid.

### Regression-test strategy

- **Source-reading guard (primary).** Extend `tests/components/BottomNavigation.slide.test.mjs` (it
  already source-reads `useKeyboard.ts`). Assert the chosen anti-flicker mechanism is present —
  e.g. a debounce timer / separate open-vs-close thresholds in `useKeyboard.ts`. Keep the existing
  `getNavYTarget` pure-function assertions unchanged.
- **Pure-function test** if hysteresis is extracted into a helper (e.g. `resolveKeyboardOpen({ heightDelta, wasOpen, openThreshold, closeThreshold })`) — contract-test that a mid-range
  `heightDelta` does NOT flip an already-open state.
- **No reliable runtime DOM test** — `visualViewport` resize sequencing is a WebView behavior the
  Node mock can't reproduce faithfully (false-green risk).

### Risks / unknowns

- **Device-only reproduction.** This is an Android WebView timing bug; it will not reproduce in
  the Node suite and may not reproduce in desktop Chrome devtools keyboard emulation. **Mandatory
  on-device UAT** (per `feedback_browser_storage_human_verify.md`). Plan must include a UAT step:
  open keyboard on Ask, confirm nav slides once smoothly with no up/down flicker.
- Tuning the debounce/hysteresis values is empirical — may need a device pass to settle.
- `useKeyboard` is global (used app-wide). A debounce change affects keyboard detection everywhere,
  not just Ask. Verify it doesn't regress other keyboard-dependent layout (e.g. ChatInput visibility).

## BUGFIX-04 — First Send tap dismisses keyboard instead of sending

### Confirmed root cause

`app/src/components/ChatInput.tsx:176-194`: the Send button is `<button type="submit">` inside a
`<form onSubmit={handleSubmit}>` (`:84-85`). It relies on the default click → form-submit path
(`onClick`/submit fires on pointer-*up*, after focus changes). On Android WebView, tapping the Send
button (outside the focused `<input>`) fires `focusout`/blur on the input *first*; the WebView
consumes that first tap to dismiss the keyboard, and the click/submit never reaches `handleSubmit`.
The user must tap a second time (now the input is already blurred) for the submit to register.

This is the classic "blur eats the first tap" pattern: any handler that fires on click/mouseup/submit
loses the race to the blur on the first interaction.

### Recommended fix approach

In `ChatInput.tsx`, make Send fire on **pointer-down** and **preserve input focus**:

- Add `onPointerDown` (or `onMouseDown`) to the Send button that calls `e.preventDefault()` (to stop
  the input from blurring and the keyboard from dismissing) and then invokes the send. Keep the
  existing `type="submit"` / `onSubmit` for Enter-key submit, but route the *tap* through the
  pointer-down handler so it runs before blur.
- Concretely: extract the send body from `handleSubmit` into a `submitMessage()` and call it from
  both `handleSubmit` (Enter) and a new `onPointerDown={(e) => { e.preventDefault(); submitMessage(); }}`
  on the Send button. `preventDefault()` on pointerdown/mousedown keeps focus on the input so the
  keyboard stays open and the message sends on the first tap.

Do NOT change the button to a non-submit type without preserving Enter-key submit, and do NOT remove
the `disabled={!canSend}` guard.

### CLAUDE.md invariants the fix must preserve

- **ChatInput flex-shrink:** keep `minWidth: 0` on the `<input>` and the Send button ≤ 44px. Do not
  grow the button. (`tests/components/ChatInput.flex-shrink.test.mjs`)
- **SwipeTabContainer onFocusOut re-snap:** if the fix preserves input focus (no blur on Send), the
  `onFocusOut` re-snap won't fire on send — that's fine (it only matters on actual keyboard close).
  Verify the fix doesn't introduce a spurious blur that triggers nav drift (ties into BUGFIX-03).
- **Root overflow clip:** unaffected, but the `preventDefault()` must not break normal input typing.

### Regression-test strategy

- **Source-reading guard (primary).** New file `tests/components/ChatInput.send-on-pointerdown.test.mjs`
  (mirrors `ChatInput.flex-shrink.test.mjs`). Assert the Send button has an `onPointerDown` (or
  `onMouseDown`) handler that calls `preventDefault()`, and that the send body is shared with the
  form submit (so Enter still works). This locks the fix against a refactor that reverts to click-only.
- **Pure-function test** if the send body is extracted into a testable `submitMessage(message, disabled)`
  helper — contract-test trimming + disabled-guard behavior.
- **No reliable runtime DOM test** for the blur-vs-tap ordering — it's a WebView event-timing behavior.

### Risks / unknowns

- **Device-only reproduction.** Blur-eats-tap is a WebView/touch behavior; desktop Chrome may not
  reproduce it. **Mandatory on-device UAT:** keyboard open → single tap Send → message sends, keyboard
  may stay open or close per design but the message must send on the first tap.
- `preventDefault()` on pointerdown can suppress the synthetic click — ensure the Send still fires
  exactly once (no double-send from both pointerdown and a later click). Test for single-fire.
- Interaction with BUGFIX-03: preserving focus on Send means the keyboard stays open after send;

## Common Pitfalls

### Pitfall 1: "Fixing" BUGFIX-03 by disabling keyboard resize
**What goes wrong:** Installing `@capacitor/keyboard` with `resize:'none'`.
**Why it happens:** It superficially stops the layout from moving.
**How to avoid:** CLAUDE.md explicitly forbids it — users rely on `adjustResize` so the input
scrolls above the keyboard. Fix the `keyboardOpen` toggle in `useKeyboard`, not the resize mode.

### Pitfall 2: Confusing text-art `textArtContent` with news `bodyMarkdown`
**What goes wrong:** Adding a "reject empty content" gate that breaks the deliberate news
empty-`bodyMarkdown` defer-to-streaming design.
**How to avoid:** The BUGFIX-02 gate applies to `textArtContent` only. News `bodyMarkdown: ''` is correct.

### Pitfall 3: Trusting a green Node suite for BUGFIX-03/04
**What goes wrong:** Phase 55 shipped 2 defects past a fully green suite (browser-storage/platform).
**How to avoid:** Source-reading guards + mandatory on-device UAT for the two keyboard bugs.

### Pitfall 4: Restructuring `useQuestions.askStreaming` for BUGFIX-01
**What goes wrong:** Touching the message array breaks Phase 35 byte-stability.
**How to avoid:** Keep the binding fix entirely in `AskScreen.tsx`. `askStreaming` is unchanged.

## Open Questions

1. **Should the keyboard stay open after Send (BUGFIX-04)?**
   - What we know: `preventDefault()` on pointerdown preserves focus → keyboard stays open.
   - What's unclear: whether the operator wants the keyboard to persist for rapid follow-up or close.

   - What we know: stripping is what exposes good posts to a bad regeneration under Gemini.
   - What's unclear: whether locale-correct regeneration is worth the corruption risk once the
     budget fix lands.
   - Recommendation: keep the strip but add the reject-empty gate so regeneration is safe; revisit
     if device testing still shows mismatched-locale text-art.

   - What we know: text-art uses `maxTokens: 80`. Session-title generation also makes a short call.
   - Recommendation: audit all `maxTokens:` call sites with small values during planning.
