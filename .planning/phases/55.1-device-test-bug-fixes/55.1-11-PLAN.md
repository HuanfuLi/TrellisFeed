<objective>
GAP-A (BUGFIX-05) — SECOND ATTEMPT, DIFFERENT APPROACH. The first attempt (55.1-05) used an eased
translateY that decays to 0; it FAILED on device (round 3): the decaying offset is applied AND removed
within the SAME paint as Android's native adjustResize reflow, so no transit is ever visible — the bar
still teleports. The decay approach is REJECTED.

New approach: drive the input bar's position CONTINUOUSLY off `visualViewport` resize/scroll events.
Android emits a STREAM of these throughout the keyboard animation. Track the live keyboard inset
(`window.innerHeight - visualViewport.height - visualViewport.offsetTop`) on each event and translate
the bar by that live value, so the bar FOLLOWS the keyboard's animated rise frame-by-frame rather than
snapping. Optionally pair with the `interactive-widget=resizes-content` viewport meta + CSS keyboard
insets as a companion lever.

Purpose: make the input reposition visibly track the keyboard, twice-burned — so the plan explicitly
notes this is attempt 2 and the decay approach is rejected, keeps the pure-helper test pattern
(reworked to compute the live offset from visualViewport geometry), and ends with a clear device-UAT
checkpoint listing the tuning knobs.

Output: reworked `chatinput-offset.ts` (live inset, no decay), the ChatInput transform wired to a
continuous visualViewport listener, all CLAUDE.md keyboard/overflow/ChatInput invariants preserved
(source-guarded), nav instant-hide unregressed, and mandatory on-device UAT.
</objective>
