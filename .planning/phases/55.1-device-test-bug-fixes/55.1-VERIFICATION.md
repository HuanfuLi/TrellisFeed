### Human Verification Required

The Node test suite proves the **structural** correctness of all four fixes. Three of the four fixes require on-device confirmation because the failure modes involve Android WebView runtime behavior that Node cannot replicate (per `feedback_browser_storage_human_verify.md`).

### 1. BUGFIX-01: Cross-Session Response Leak (Rapid Switch Race)

**Test:** On a physical Android device (or the configured emulator), open the Ask screen. Send a question that triggers a long LLM response (e.g., ask for a detailed explanation). While the answer is still streaming, immediately tap "New Chat" and ask a second question. Repeat the cycle: send a long question, open new chat, send again — do this 3-5 times quickly.
**Expected:** No prior session's answer ever appears under a different session's question. Each session only shows the answer that was requested in that session, even if the answer from a slower session finishes after you've moved on.
**Why human:** The Node suite proves the structural fix — `originSessionId` is captured, abort is wired, the finally block guards on controller identity. But the actual cross-session render is a live async race between streaming I/O and React state transitions that cannot be replicated without a real LLM provider and real async timing.

### 2. BUGFIX-02: Text-Art Truncation on Provider/Locale Switch (Live Gemini UAT)

**Test:** With a Gemini thinking model (e.g., gemini-2.5-flash) configured and a live API key: (a) generate some text-art posts so several have full sentences visible; (b) switch the LLM provider to the Gemini model; (c) change the locale (e.g., EN → ZH → EN). Observe all existing text-art cards.
**Expected:** (1) Existing text-art posts keep their full sentence content — no card collapses to "T", "Is your", or another fragment. (2) No Gemini API error appears in the console from the `thinkingConfig.thinkingBudget: 0` field.
**Why human:** The Node suite locks the structural fix (gate, merge guard, budget 512, thinkingConfig). Confirming the live Gemini API accepts the `thinkingConfig.thinkingBudget` field and that no content regression occurs requires a real API key and the actual Gemini thinking model response path.

### 3. BUGFIX-03: Bottom-Nav Flicker on Keyboard Open

**Test:** On an Android device or emulator, navigate to the Ask screen. Tap the text input to open the keyboard.
**Expected:** The bottom navigation bar slides down smoothly in a single motion. There is no oscillation (it does not slide down, then briefly up, then down again). The ChatInput remains visible and usable. Other screens that use `useKeyboard` (the hook is global) should also not regress — check that the Planner and Settings screens' keyboard interactions are unaffected.
**Why human:** The hysteresis contract is provably correct (14/14 pass), but the flicker is an Android WebView `visualViewport.resize` animation-timing phenomenon. Whether the open=150 / close=80 thresholds are empirically correct for this specific device's WebView requires on-device measurement. Threshold tuning (if residual oscillation occurs) is done by adjusting the exported constants in `keyboard-hysteresis.ts`.

### 4. BUGFIX-04: Send Fires on First Tap

**Test:** On an Android device or emulator, navigate to the Ask screen. Tap the text input to open the keyboard. Type a message. Tap the Send button exactly once.
**Expected:** (1) The message sends on the first tap — you do not need to dismiss the keyboard first. (2) Exactly one message is sent (no double-send). (3) The keyboard remains open after the send (Open Question 1 default).
**Why human:** The Node suite proves `onPointerDown+preventDefault` is wired and `submitMessage` is shared. The actual fix depends on `preventDefault` suppressing the synthetic click and running before the blur event in Android WebView's event delivery order — this is device/OS/WebView-version-dependent behavior that Node cannot replicate.
