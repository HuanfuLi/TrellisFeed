# Phase 55.1 Plan 06: GAP-B Trellis Perf at Scale Summary

Profiled the Planner trellis at production scale, found the in-loop blossom-date
localStorage persistence to be the dominant cost (O(N^2) churn — 54.24 ms of a
60.87 ms build at N=3000), and batched it to a single flush — cutting full
`buildTrellisState` from 60.87 ms to 6.72 ms (~9x) at N=3000.

## What Was Built

**Task 1 — Profiling artifact (MEASURE FIRST).** `app/scripts/profile-trellis.mjs`
seeds synthetic graphs (N ∈ {200, 1000, 3000} anchors, +3 QA children each, 30%
blossom, 12 branches) and times each phase of `buildTrellisState` in isolation
under `node --test`: full build, aggregate `computeLeafState`, fcMap build, the
in-loop blossom-date persistence side-effect, and an 8-event recompute storm.
The artifact names the dominant phase explicitly.

**Task 2 — Targeted fix + perf-invariant test.** `buildTrellisState` now mutates
the in-memory `blossomDates` map during the per-node loop and flushes ONCE after
the loop via a single `replaceBlossomDates(blossomDates)` call, replacing the
prior per-node `setBlossomDate`/`clearBlossomDate` calls (each a full
`getBlossomDates()` JSON.parse + `setItem` JSON.stringify of the growing map).
`tests/services/trellis-perf.test.mjs` executes the logic and asserts the
invariant: 300 blossom anchors ⇒ exactly ONE blossom-dates write; idempotent
recompute ⇒ zero writes; leaf-state outputs unchanged.

`blossomWrite` (in-loop persistence) is **89%** of the 60.87 ms build at N=3000.
`computeLeafState` aggregate (2.06 ms) and fcMap build (~0 ms) are negligible by
comparison. The fix therefore targets ONLY the blossom-write phase.

**Reading the `blossomWrite` column post-fix (60.05 ms):** the profiler's
`blossomWrite` measurement *intentionally* reproduces the OLD per-node
`setBlossomDate`/`clearBlossomDate` pattern in isolation — it does NOT call the
fixed `buildTrellisState`. It is kept as the standing measurement of exactly how
much cost the fix removed from the hot path. The meaningful before/after is the
`build` column (the real, now-fixed function): 60.87 → 6.72 ms.

## How It Works (the fix)

`app/src/services/trellis-state.service.ts`:
- Replaced the `setBlossomDate, clearBlossomDate` imports with `replaceBlossomDates`.
- Added a `let blossomDatesDirty = false` flag before the node loop.
- In both the anchor branch and the legacy-node branch, the set/clear logic now
  mutates the in-memory `blossomDates` object and flips `blossomDatesDirty = true`
  instead of touching localStorage.
- After the loop: `if (blossomDatesDirty) replaceBlossomDates(blossomDates)` — a
  single `setItem` write.

Outputs are byte-identical to the prior implementation: the local `blossomDates`
map is mutated the same way (same `today()` values written, same keys deleted),
so each node's `blossomSinceDate` and `leafState` are unchanged. Only the
persistence is batched.
