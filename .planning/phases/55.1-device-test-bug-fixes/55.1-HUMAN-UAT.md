### 5. GAP-A (BUGFIX-05) — ChatInput animates smoothly to keyboard-open position (no teleport)
expected: On the Ask screen (Android), tap the text input. The input bar should SLIDE/settle into its keyboard-open position over a short ease, not instantly jump/teleport.
result: **FAILED again (round 4, 2026-05-22).** Same teleport behavior. Two approaches now failed (55.1-05 eased-decay, 55.1-11 visualViewport-residual-follow). **Root cause identified (code-level, round 4):** under Android `adjustResize` the WebView LAYOUT viewport shrinks instantly in a SINGLE paint when the IME shows — there are no intermediate geometry frames for `visualViewport` to report. So 55.1-11's `residual = liveVisualInset − layoutReflow` collapses to ~0 (both jump together) → no transform → teleport. 55.1-05 failed because its invert+restore happened in the SAME paint (browser coalesces to final state). Neither did a proper TWO-FRAME FLIP. **RESOLVED — SKIPPED / won't-fix (round 5, 2026-05-22).** Research (operator-requested: "how does WeChat do it") confirmed a smooth follow needs the IME animation's per-frame progress, exposed ONLY natively (Android `WindowInsetsAnimation.onProgress`, API 30+ — WeChat's chat input is a native view). The web platform can't reach it from a WebView: `visualViewport` resize fires ONCE at the END of the animation (no per-frame stream), and the VirtualKeyboard API / `env(keyboard-inset-*)` is unsupported in Android WebView. Robust fix = a native Capacitor plugin bridging `WindowInsetsAnimation.onProgress` → out of scope ("not minor risk"). Per operator: skip. Both attempts (55.1-05, 55.1-11) REVERTED to clean baseline (commit 9e33efad); the teleport is accepted. BUGFIX-03/04 unaffected. See memory `feedback_keyboard_follow_webview_impossible`.

### 6. GAP-B (BUGFIX-05) — Planner trellis stays smooth at scale
expected: Smooth swipe + feed scroll with a large trellis.
result: **FAILED again (round 4, 2026-05-22).** Still very laggy swiping Home↔Planner↔Ask in Dev mode, and still laggy on Home feed vertical scroll (Planner fully off-screen). **Root cause identified (code-level, round 4):** 55.1-09's `animationsEnabled` gate only changed framer-motion animation TARGETS to static — it left ALL the machinery mounted and painting: ~93 leaf `motion.g` VisualElements (3 per leaf × 31) + ~60 vine `motion.path` + a per-leaf `filter: drop-shadow(...)` on every leaf, ALL always-mounted regardless of route. framer-motion VisualElements carry per-element overhead and SVG `filter` forces a rasterization pass per leaf, so the giant off-screen canvas keeps starving the compositor. REOPENED → robust fix: when `animationsEnabled === false`, render leaves/vines as PLAIN SVG (`<g>`/`<path>`, no `motion.*`, no `useAnimationControls` usage, no `drop-shadow` filter) so the off-screen canvas mounts ZERO framer-motion VisualElements and runs ZERO filter passes. **RE-FIXED (round 5, 2026-05-22, commit c08c30f2) — pending device re-test.** When `animationsEnabled === false`: TrellisLeaf early-returns a plain `<g>` (no motion.g/useAnimationControls/drop-shadow), TrellisCanvas renders vine stems as plain `<path>` (motion props spread only when animating). Foreground branch byte-identical. Source-guard: `tests/components/trellis-static-offscreen.test.mjs`.
