Migrate the whole heavy-store layer off localStorage to SQLite-primary (D-09..D-13). If NO-GO (OPFS unavailable in dev), keep LocalStorageBackend as the ACTIVE browser backend and ship WASMSQLiteBackend gated behind the OPFS try/catch fallback — the inversion + Float32 BLOB + cutover work all still land against whichever backend is active.

| npm registry -> build | `@sqlite.org/sqlite-wasm` in the dependency tree (supply chain) |

| async delete vs cold restart | A row deleted in the mirror but not yet flushed to SQLite could resurrect on hydrate |

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-55-05b | Denial of Service | OPFS unavailable -> app crash | mitigate | WASMSQLiteBackend.init() throws on OPFS failure; getDB catch falls back to LocalStorageBackend with a logged warning (Pitfall 1) — app never crashes on a backend-init failure. |
| T-55-05c | Information Disclosure | OPFS SQLite file | accept | Local-first, origin-scoped, single-user; no PII egress. Low risk. |
