## Pattern Chosen — Why (a) Over (b)/(c)

CONTEXT Specifics offered three candidate fix patterns for G2:

| Pattern | Description | Diff size | Tradeoff |
|---------|-------------|-----------|----------|
| **(a) — chosen** | `pointer-events:none` on overlay, `auto` on close button | ~10 lines | Pure declarative CSS, easy to revert, preserves all gesture-detection wiring |
| (b) | Single-fire tap detector that fires once then removes itself | ~30+ lines | Custom event-listener lifecycle code, harder to reason about iOS Safari quirks |
| (c) | Two-zone overlay: top 80% transparent passthrough, bottom 20% intercepts | ~20 lines | User-confusing UX (tap-here-but-not-there split) |

**Pattern (a) was chosen** because:
1. It is the minimum-diff option that satisfies D-05 (overlay reduced, not removed — DOM structure preserved)
2. The `videoPlaying` state owner sits in the parent (`ConceptCardList` / `InlineInfoFlow`) and the swipe-stop listener is wired via `SwipeTabContext` at `InfoFlow.tsx:927-944` — totally independent from the overlay's onClick. Removing the overlay tap interception does NOT break D-29 swipe-stop.
3. Per CONTEXT D-07: even if YouTube native controls don't reach the user (web target works; on-device iOS may behave differently), the visible close button is the explicit stop affordance. Acceptable failure mode.
4. If pattern (a) breaks on device (e.g., close button no longer hittable due to z-index re-stacking), reverting is a 6-line CSS revert vs. 30+ lines for pattern (b).

## D-07 Fallback Contingency

CONTEXT D-07 explicitly accepts a fallback if pattern (a) doesn't surface YouTube's native controls on iOS Safari:

> "Tap-to-restart-playback is acceptable as a fallback. If reaching YouTube's native controls is impossible without breaking gesture detection, the minimum acceptable outcome is: tapping a playing video does nothing (no restart), and a clear stop button is visible."

After pattern (a):
- **Best case:** taps reach YouTube → native pause/scrub/fullscreen overlay shows → user controls playback inline. Close button still works. ✓ Full success
- **Acceptable case (D-07):** taps reach YouTube but iOS Safari's iframe sandbox quirks prevent the native pause overlay from showing. Tap on playing video does nothing (no restart). Close button still visible and functional. ✓ Acceptable per D-07
- **Failure case (would trigger pattern (b) escalation):** removing the onClick on the overlay somehow re-introduces the original symptom ("tap restarts playback"). This shouldn't happen because the iframe + YouTube embed is what's now receiving taps directly, not the overlay — but if it does, escalate to pattern (b) single-fire tap detector.

The `retest_note` in 31-UAT.md is written as a CONTINGENCY — pending operator on-device confirmation. If operator reports failure case, this plan will need an adjustment commit re-applying pattern (b) and updating the retest_note.
