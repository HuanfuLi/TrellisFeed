## Root Cause

**Symptom (operator-confirmed):** After Phase 31-10 added the empty-state guard, fresh installs with `questions.length === 0` returned `STARTER_POSTS` (concept-feed.service.ts:1030 pre-fix). After the user tapped a starter post and navigated back, all 3 posts vanished from /home.

**Root cause path:**
1. `getDailyPosts([])` returned `STARTER_POSTS` BUT did **not** write them to the daily cache.
2. `HomeScreen.tsx:130-134` re-syncs `dailyPosts` from `getCachedDailyPosts()` whenever `location.pathname === '/home'` (i.e., every nav back).
3. `getCachedDailyPosts()` reads from the cache. Cache had no posts (starters were never written). Result: `dailyPosts = []`.
4. **Even worse:** any subsequent `getDailyPosts(questions)` call where `questions.length > 0` hits the empty-state branch with `if (questions.length === 0)` returning false → returns `[]` — starters lost forever once a single question is created via the Welcome CTA.

**Fix path (per CONTEXT D-11/D-12):**
- D-11: Persist `STARTER_POSTS` to cache on cold-start so the cache-hit branch returns them on subsequent navigations.
- D-12: Decay starters once cache contains 3+ organic posts (otherwise they'd persist forever).
- D-13: Verify `PostDetailScreen` renders starter content; add guard if needed.

## Fix Implementation

### A. New helper module: `app/src/services/starter-posts-decay.ts`

Pure helper (no transitive deps). 4 exports:

| Export | Type | Purpose |
|---|---|---|
| `STARTER_POST_IDS` | `Set<string>` | Lockstep registry of the 3 starter IDs (`starter-welcome`, `starter-knowledge-growth`, `starter-daily-feed`) — kept in sync with `STARTER_POSTS` array in concept-feed.service.ts:54-79 |
| `isStarterPostId(id)` | `(string) => boolean` | Membership check |
| `shouldDecayStarters(posts)` | `(Array<{id}>) => boolean` | Returns true when `posts.filter(p => !STARTER_POST_IDS.has(p.id)).length >= 3` |
| `filterDecayedStarters(posts)` | `<T extends {id}>(T[]) => T[]` | Returns posts unchanged if not decayed; else drops starters |

Same extraction rationale as `concept-feed-dedup.ts` (Phase 32.1-02): importing `concept-feed.service.ts` under plain `node --test` triggers the i18next JSON-import-attribute failure chain (`graph.service` → `planner.service` → `locales/en.json`). Tests assert the helper's exported contract instead.

## G6 Cold-Start Flicker — Expected Side-Effect Closure

CONTEXT lists G6 (cold-start loading flicker) as out-of-scope for Phase 32.1, BUT the persist-to-cache change in this plan is expected to **eliminate it as a side effect**. Mechanism:

- **Before:** Cold-start path → `getDailyPosts([])` returns `STARTER_POSTS` synchronously, but cache empty → next /home re-sync via `getCachedDailyPosts()` returns `[]` → flicker (3 posts → 0 → 3 again on async re-fetch).
- **After:** Cold-start writes cache → next /home re-sync via `getCachedDailyPosts()` returns the same 3 starter posts from cache → no flicker.

If the operator's next device-build no longer reports the G6 flicker, this plan can be cited as the closing fix and G6 can be marked closed in subsequent docs.
