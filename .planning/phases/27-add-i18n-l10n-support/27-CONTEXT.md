# Phase 27: Add i18n/L10n support - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Introduce internationalization and localization plumbing across EchoLearn so UI chrome, date/number formatting, LLM-generated content, and TTS voice all respect a user-selected locale. Ship with English (canonical) + Simplified Chinese + Spanish + Japanese bundles fully translated. Locale is auto-detected on first launch, confirmed during onboarding, switchable instantly from Settings, and persisted in `UserPreferences`. No RTL.

The phase delivers working plumbing + all four complete locale bundles; it does NOT retroactively translate user-generated content or mindmap node labels stored in a prior locale.

</domain>

<decisions>
## Implementation Decisions

### Target locales

- **D-01:** Ship 4 locales: EN (source/canonical) + ZH (Simplified Chinese) + ES (Spanish) + JA (Japanese). No RTL locales in this phase.
- **D-02:** EN bundle is hand-authored. ZH/ES/JA bundles are generated by a dev-time Sonnet subagent (Task tool, spawned locally during execution) and committed as hardcoded JSON — **never translated at runtime**. See "Translation workflow" below.
- **D-03:** Phase completes with all 4 bundles fully translated. No empty stubs, no partial coverage. Every key that exists in `en.json` must exist in `zh.json`, `es.json`, and `ja.json` before merge.

### Library & structure

- **D-04:** Use **`react-i18next`** (+ `i18next` core). ~14KB gz, Suspense-aware, familiar to downstream subagents.
- **D-05:** Flat JSON bundle per locale: `app/src/locales/en.json`, `zh.json`, `es.json`, `ja.json`. Each file is a nested object grouped by feature namespace: `home.*`, `planner.*`, `ask.*`, `review.*`, `podcast.*`, `settings.*`, `onboarding.*`, `posts.*`, `graph.*`, `common.*` (buttons/toasts/errors shared across screens).
- **D-06:** Type-safety: a small build-time script generates a TypeScript union of every key from `en.json` (or use react-i18next's [typescript augmentation](https://www.i18next.com/overview/typescript)). Typos in `t('home.titlee')` must fail TypeScript compile.

### Translation workflow (critical rule)

- **D-07:** **Runtime LLM calls for translation are prohibited.** The app's `llmProvider` (Claude/Gemini/OpenAI) must never be invoked to translate UI copy at runtime.
- **D-08:** Translation flow when new strings are added:
  1. Developer adds new key(s) to `en.json` with canonical English copy.
  2. Developer runs a documented script (`scripts/translate-locales.mjs` or similar) or invokes a Sonnet subagent via the Task tool with instructions to fill the missing keys in `zh.json`, `es.json`, `ja.json`.
  3. Developer reviews the generated translations (human-in-the-loop).
  4. Developer commits all 4 bundle files in one PR.
- **D-09:** Bundle files must be tracked in `CLAUDE.md` (project root) so anyone modifying a screen/component is reminded to also update matching locale entries. Add a dedicated section "i18n / Locale Bundles" listing every bundle path and the canonical EN-first workflow rule.

### Translation scope

- **D-10:** UI chrome: every hardcoded user-visible English string in `app/src/screens/**`, `app/src/components/**` (including `BottomNavigation` tab labels, button text, form placeholders, empty-state copy, toast messages, error strings, the entire `OnboardingScreen` flow, every `SettingsScreen` row label + description).
- **D-11:** Date/number/time formatting: refactor `app/src/lib/date.ts`. Replace hardcoded `'en-US'` locale tag in `formatDate` and `formatDateLabel` with the current i18n locale. `getGreeting()` returns a translated greeting ("Good Morning" → "上午好" / "Buenos días" / "おはよう") via `t('common.greeting.morning' | '...afternoon' | '...evening')`.
- **D-12:** LLM output locale: `app/src/providers/llm.ts` `chatCompletion()` (and any streaming variant) reads the current locale from settings and **centrally** prepends a `Respond in {localeName}` system message to every request. All call sites (useQuestions, question.service, concept-feed, news, podcast, flashcard, classification) inherit automatically — no per-site changes required. Locale is surfaced as a human-readable name, not a BCP-47 tag (e.g. "Simplified Chinese", "Spanish", "Japanese", "English").
- **D-13:** Podcast TTS: `app/src/providers/tts.ts` `synthesize()` reads the current locale and passes a locale-appropriate voice ID (mapping: `en` → existing default, `zh` → a Mandarin voice, `es` → a Spanish voice, `ja` → a Japanese voice). Specific voice IDs per provider TBD during research/planning.
- **D-14:** YouTube feed (`app/src/services/youtube.service.ts`): pass `hl` (interface language) + `regionCode` matching the current locale on Data API v3 calls so non-EN users get locale-native video results.
- **D-15:** Web search (`app/src/services/web-search.service.ts`): **intentionally stays English / global**. EN has broader coverage for learning content and citations. Do NOT pass locale to Tavily.
- **D-16:** User-generated content (stored ChatSessions/Questions/Posts, Anchor/Cluster labels, flashcard cards) stays in its original language after a locale switch. UI chrome switches; stored content does not. No retroactive translation, no per-item "Translate" button in this phase.

### Locale switching UX

- **D-17:** First-launch detection: read device language via `navigator.language` (web fallback) and `@capacitor/device` `Device.getLanguageCode()` (native). If the detected code matches a supported locale, use it as the default. Otherwise default to EN.
- **D-18:** Onboarding confirmation: `OnboardingScreen` gains a language-confirmation step (label shown in all 4 languages: "Language / 语言 / Idioma / 言語") letting the user confirm or override the auto-detected locale before the rest of onboarding proceeds.
- **D-19:** In-app switcher: new `SettingRow` at the top of `SettingsScreen` (above current first row). Tapping an option calls `i18n.changeLanguage(locale)` + persists to `mockSettingsService`. **Instant switch** — every screen re-renders via `useTranslation` hooks. No app restart.
- **D-20:** Persistence: new `locale: 'en' | 'zh' | 'es' | 'ja'` field on the existing `UserPreferences` type, stored via `mockSettingsService` in localStorage with the rest of the preferences blob.
- **D-21:** Missing-key fallback: `react-i18next` configured with `fallbackLng: 'en'`. Missing keys render the EN string silently in production; console warning in dev. Must never render the raw key path to the user.
- **D-22:** Mid-stream LLM handling: if the user switches locale while a chat completion is streaming, **discard the in-flight stream**. The new locale takes effect on the next request. Do not attempt to re-translate mid-flight or continue in the old locale.
- **D-23:** Font stack: CSS variable `--font-sans` (or equivalent) is swapped based on the active locale via a body-level `data-locale` attribute or a root class.
  - `en` / `es`: current stack (Latin-only).
  - `zh`: current stack + append `'PingFang SC', 'Hiragino Sans GB', 'Microsoft YaHei', sans-serif`.
  - `ja`: current stack + append `'Hiragino Sans', 'Yu Gothic', 'Meiryo', sans-serif`.
  - System fonts only — NO webfont downloads. Keep bundle small.
- **D-24:** Visual QA: UAT includes a manual walkthrough of every first-level screen + key sub-screens in each of the 4 locales, with screenshots archived. Must verify no overflow (ES ~20% longer than EN) and no awkward breaking (JA). Detailed UAT task list to be produced in planning.

### Claude's Discretion

- Exact voice IDs per TTS provider for ZH/ES/JA — research during planning.
- Whether to use `i18next-browser-languagedetector` plugin or a small custom detector that integrates with `mockSettingsService`. Either is acceptable; custom detector may be cleaner given UserPreferences persistence.
- How to key-extract existing hardcoded strings (manual pass vs ESLint custom rule vs AST codemod). Planner decides based on LoC estimate.
- How strictly to namespace (one flat `en.json` vs lazy-loaded namespaces). Start with a single file; split only if bundle size becomes a measurable concern during planning.
- Exact shape of the Sonnet-subagent translation script — planner/executor designs the invocation.

<specifics>
## Specific Ideas

- Switcher label in Settings is shown in all four languages side-by-side: "Language / 语言 / Idioma / 言語" so every user recognizes it regardless of current locale state (mitigates the "stuck in a locale I can't read" failure mode).
- The translation-generation Sonnet subagent should be invoked with an explicit instruction template that: reads `en.json`, reads the existing target locale file for keys already translated (to preserve manual edits), produces only the diff for missing keys, and returns a full merged locale file. Exact template design is a planner/executor concern.
- `CLAUDE.md` must gain an "i18n Workflow" section stating: (1) canonical EN first, (2) every new key must land in all 4 bundles in the same PR, (3) path list of bundle files, (4) instruction to spawn a Sonnet subagent for non-EN translations, (5) reminder that runtime LLM translation is forbidden.

</specifics>

<deferred>
## Deferred Ideas

- **RTL/Arabic support** — would require CSS logical-property migration (padding-inline, margin-inline, text-align: start/end) across all inline-style sites. Separate phase.
- **Retroactive content translation** (per-item "Translate" button on old Q&As/posts/mindmap labels) — violates dev-time-only translation rule unless pre-generated at creation time; defer to a follow-up phase with its own design.
- **Pseudo-locale for dev overflow detection** — automated layout-break catcher; nice-to-have.
- **Extra locales** (ZH-Hant, KO, DE, FR, etc.) — deferred to demand-driven follow-ups.
- **Locale-specific font downloads (webfonts)** — would improve typography on Linux / rare OSes but bloats bundle; ship system-font-only for now.
- **Custom TTS per locale** — currently using a single provider; future phase could add provider-per-locale routing for higher-quality voices.
- **Markdown/KaTeX content in RTL mode** — n/a for this phase; revisit with RTL.
- **Automated translation QA (e.g., screenshot diff per locale)** — deferred to tooling phase.
