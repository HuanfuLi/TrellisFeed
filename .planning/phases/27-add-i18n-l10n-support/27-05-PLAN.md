  <action>
    For each of the 8 screens listed, perform this pattern:

    Step 1: Identify all user-visible English strings. Scan for:
    - JSX text nodes (between opening/closing tags)
    - `placeholder=` attributes
    - `aria-label=` attributes (if user-visible via screen reader or tooltip)
    - `alt=` attributes with real text (skip `alt=""`)
    - Button/label/title props
    - `toast('...')` calls (look for toast('English message') — replace with toast(t('namespace.key')))
    - Error strings displayed in UI
    - Empty-state messages

    DO NOT extract:
    - Strings passed to LLM (system prompts — D-07 forbids runtime translation of those, AND they stay EN by design so the LLM understands)
    - Strings used as localStorage keys, event names, or internal identifiers
    - Strings used as CSS values (font-family lists, color names)
    - Proper nouns: "EchoLearn", "OpenAI", "Claude", "Gemini", "YouTube", "Tavily", "API", "TTS", "LLM", "SM-2"
    - Developer-facing console.log/error/warn strings
    - Numeric/template strings that are structurally formatted (e.g. dates — use existing date.ts helpers)
    - **Cross-locale branded labels** (D-18/D-19) — explicitly these strings in OnboardingScreen.tsx (added by Plan 03) and SettingsScreen.tsx (added by Plan 04 AFTER this plan runs, but the convention applies everywhere):
      - `"Language / 语言 / Idioma / 言語"`
      - `"English · 中文 · Español · 日本語"` (or similar multi-script headers)
      - `"Continue · 继续 · Continuar · 続ける"`
      - The per-option script-native labels inside the language picker buttons (`"English"`, `"简体中文"`, `"Español"`, `"日本語"`)
      These are intentionally hardcoded so users stuck in an unreadable locale can still find the switcher. Leave them verbatim.

    Scope discipline (mirrors Plan 06 Task 2): if this task's grep surfaces >100 new keys across the 8 screens combined, PAUSE execution, commit what is done, and spawn a gap-closure plan for the overflow screens rather than blowing the task context budget. ~10-30 keys per screen is the healthy range; >50 on a single screen likely means additional sub-namespacing is warranted.

    Step 2: Add `useTranslation()` at the top of each screen function:
    ```typescript
    import { useTranslation } from 'react-i18next';

    export function HomeScreen() {

    }
    ```

    Step 3: Replace each hardcoded string with `t('<namespace>.<key>')`. Use the screen's namespace:
    - HomeScreen → `home.*`
    - PlannerScreen → `planner.*`
    - AskScreen → `ask.*`
    - ReviewScreen → `review.*`
    - GraphScreen → `graph.*`
    - PodcastScreen → `podcast.*`
    - SettingsScreen → `settings.*` (preserving the intended Plan-04 switcher labels which intentionally stay in all 4 scripts; note that the Plan 04 switcher row itself will be inserted AFTER this plan per dependency chain)
    - OnboardingScreen → `onboarding.*` (preserving the Plan-03 language step labels in all 4 scripts — those stay hardcoded per D-18)

    For strings that are genuinely shared (e.g. "Cancel", "Save", "Today"), use the `common.*` namespace. Plan 01 seeded `common.today`, `common.cancel`, `common.save`, `common.delete`, `common.greeting.*` — extend as needed.

    Key naming convention:
    - Flat nested structure: `home.feedEmpty`, `planner.suggestedMoves`, `ask.placeholder`
    - Descriptive, not copy-pasting English: prefer `review.cardFrontPrompt` over `review.showQuestion`
    - For buttons: `*.action.<verb>` (e.g. `podcast.action.generate`)
    - For interpolation: name the variable descriptively — `t('home.greeting', { name })` with en.json `"greeting": "Welcome, {{name}}"`

    Step 4: Write all new keys to `app/src/locales/en.json`. After each screen, validate the bundle:
    ```bash
    cd app && node -e "JSON.parse(require('fs').readFileSync('src/locales/en.json','utf8'))"
    ```

    Step 5: Mirror every new key into `zh.json`, `es.json`, `ja.json` with the EN value duplicated (parity stub). These will be translated by Plan 07's Sonnet subagent. After each screen, run parity test to catch any missed key:
    ```bash
    cd app && node --test tests/locales/bundle-parity.test.mjs
    ```
    MUST pass — if it fails, find the missing key and sync all 4 bundles.

    Step 6: Run build + tsc after each screen to catch any type errors from the augmented t() keys (if a key doesn't exist in en.json, the module augmentation from Plan 01 will make tsc fail):
    ```bash
    cd app && tsc -b --noEmit
    ```

    Step 7: Smoke test: `npm run dev`, visit each screen in turn. Confirm no raw English strings remain in the UI (everything should still SHOW English because bundles are placeholder-parity EN, but the RENDER PATH now goes through t()).

    Step 8: COMMIT: `feat(27-05): extract UI strings from 8 first-level + secondary screens → en.json (D-10 part 1)`
