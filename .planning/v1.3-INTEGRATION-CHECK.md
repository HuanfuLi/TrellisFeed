### Seam 3 — Phase 21 → 22 → 27: Locale switch mid PostDetailScreen stream

**Status: PASS (with note)**

`PostDetailScreen.tsx:166-258` uses a local `aborted` boolean ref (not `AbortController`) in the on-enter essay generation effect. When `LOCALE_CHANGED` fires, it does NOT abort `generatePostEssay` mid-stream. The `aborted` ref is only flipped on `useEffect` cleanup, which fires if the user navigates away from the post.

This means: if a user is on a PostDetail while it is streaming and switches locale via Settings, the stream continues in the old locale until the user navigates away from the post.

By contrast, `useQuestions.ts` (AskScreen, Phase 27 D-22) uses an `AbortController` wired to `LOCALE_CHANGED` — these are distinct flows. The discrepancy is not a blocker because:
1. PostDetailScreen is a sub-screen overlay, not the swipe strip — the user must navigate away to reach Settings.
2. D-22 is explicitly scoped to `askStreaming`, not post-essay generation.

The `aborted` local flag (not an `AbortController`) does guarantee the streaming result is NOT persisted after the component unmounts (navigation away), so no stale-locale essay is saved if the user immediately navigates back.

Evidence:
- `PostDetailScreen.tsx:178` — `let aborted = false;`
- `PostDetailScreen.tsx:256` — `return () => { aborted = true; };` (cleanup only)
- `useQuestions.ts:121` — `const unsubLocale = eventBus.subscribe('LOCALE_CHANGED', ...)`
- `post-essay.service.ts` — no `signal` param threading; no `LOCALE_CHANGED` subscription

Affected requirements: POST-01..08, Phase 27 D-22 (D-22 explicitly only governs `askStreaming`)

### Seam 4 — Phase 22 → 26: Swipe pan vs. Trellis scissors prune button

**Status: PASS**

Evidence:

- `SwipeTabContainer.tsx:116` — `gestureBlockedRef.current = !!((_e.target as HTMLElement)?.closest?.('[data-no-swipe-nav]'));`

### Seam 6 — Phase 23 → 27: LOCALE_CHANGED aborts Ask stream but not classification

**Status: PASS (documented behavior, acceptable)**

When `LOCALE_CHANGED` fires during `askStreaming`, the `AbortController` aborts the `chatStream` call, exits the streaming loop, and returns `null` from `askStreaming` — `buildAndSave` is never called, so `classifyAndAnchorIncremental` is never reached (lines 253-273 are skipped). The classification call at line 273 only fires if `buildAndSave` succeeds AND `filterQuestion` returns unflagged. An abort before line 247 entirely prevents classification from running.

The only edge case: if `LOCALE_CHANGED` fires after the guard at line 247 but before line 273 (during `filterQuestion` async call), classification could be invoked. However, `filterQuestion` is a single `chatCompletion` call without an abort signal. In this narrow race, classification fires once in the old locale — acceptable because: (a) classification output is not user-visible; (b) `classifyAndAnchorIncremental` uses `applyLocaleDirective` via `chatCompletion`, which reads current `i18next.language` at call time, so if locale has already changed by then, the directive is already in the new locale.

Evidence:
- `useQuestions.ts:247` — abort guard before `buildAndSave`
- `useQuestions.ts:273` — classification only reachable after guard passes
- `providers/llm/index.ts:62` — `applyLocaleDirective` is called fresh per `chatCompletion` call

Affected requirements: PIPE-01..07, Phase 27 D-22

## 4. Tech Debt

**TD-01 (WARN):** `plannerAutoGen.service.ts:115` and `concept-feed.service.ts:759` both call `defaultStrategy.computeHints(signals)` without `checkInSignals`. Phase 20's `plannerService.getRecentSignals()` API exists and is tested, but is not threaded into either call site. The `curiosityTopics` from diagnostic check-ins (Phase 20 ORCH-03) silently have no effect on suggestion ranking or feed interleaving. Fix is a 2-line change per call site.

**TD-02 (NOTE):** `PostDetailScreen.tsx` on-enter essay generation uses a local `aborted` boolean instead of an `AbortController`. This means: (a) a locale switch while a post essay is streaming does not halt the generation mid-stream; and (b) there is no timeout signal on `generatePostEssay` — a hung chatStream yields no user-visible error until component unmount. The existing `onEnterError` state handles LLM errors but not AbortErrors. Low priority since the sub-screen overlay pattern means locale changes are unlikely while a post is streaming.

**TD-03 (NOTE):** `classifyAndAnchorIncremental` in `canonical-knowledge.service.ts:742` does not accept an `AbortSignal`. In the narrow race window between the abort guard at `useQuestions.ts:247` and the classification call at line 273 (during `filterQuestion`), a LOCALE_CHANGED event will cause classification to proceed. Classification output is not user-visible and the next `chatCompletion` call will use the updated locale directive, so this is not a user-facing bug, but future hardening should thread a signal through.
