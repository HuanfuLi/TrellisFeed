## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user question text → §17.2 LLM prompt | untrusted participant input enters an LLM instruction context |
| LLM JSON → canonical user_questions + personal graph | untrusted model output becomes persisted study data |
| revised Q&A record → backend | extraction fields cross the wire into the researcher store |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-13 | Tampering | prompt injection via question text | high | mitigate | applyUserContentBracketing delimiters + untrusted-data instruction + jsonMode + strict parse; the frozen-ID allowlist is the hard enforcement layer — injected instructions cannot mint IDs |
| T-03-14 | Tampering | LLM returns unknown/cross-topic IDs | high | mitigate | Syntactic + semantic validation against same-topic repository sets before any persistence; ambiguous alias → reject (OWASP input-validation pattern) |
| T-03-15 | Tampering | double-applied extraction after retry | medium | mitigate | Job status transitions + graphMemoryService contribution keys make re-processing idempotent |
| T-03-16 | Information disclosure | extraction fields on the wire | medium | mitigate | Closed field-allowlist on wire contract and backend validation; only the four named fields added; §14.2 exclusions untouched |
</threat_model>
