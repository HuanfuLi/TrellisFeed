<objective>
Close GAP-5 (major, pre-existing, operator-approved fold-in): switching the AI provider wipes the entered API key because the provider `onChange` at `SettingsAIScreen.tsx:138-147` spreads `defaults[p]` which hard-sets `apiKey: ''`, and the config stores a single `apiKey` slot — so switching away blanks the key and switching back does not restore it.

Root cause confirmed in 52-UAT.md GAP-5. Pre-existing (introduced commit 6bdd3f4a, NOT Phase 52). Same pattern exists for the embedding-provider selector (lines 224-235). TTS provider switch (lines 348-358) also blanks apiKey but TTS has the LLM-key fallback for OpenAI — handle it consistently but do not break effectiveTtsApiKey.

Purpose: remember per-provider keys so each provider restores its own key on switch-back.

Output:
- Additive optional `apiKeys?: Record<string, string>` on LLMConfig and EmbeddingConfig (no migration framework, per CLAUDE.md `feedback_no_normalize_for_optional_fields` — read-site fallback only).
- SettingsAIScreen provider onChange: before switching, stash the current entered key under the OLD provider in `apiKeys`; after switching, restore `apiKeys[newProvider] ?? ''` into `apiKey`.
- Regression test.
</objective>
