<objective>
Close gaps G1 and G3 from 50-UAT.md Test 2.

**G1 (blocker)** — After tapping "Save name" in the inline-create flow, the new collection does not appear in the picker until the user refreshes the page. Mobile Capacitor users CANNOT refresh — operator has flagged this "dozens of times". The sheet's collections list is captured at mount via `useMemo(() => collectionService.getCollections(), [postId, open])` and never re-reads when the service mutates.

**G3 (major)** — When long-press → "Save to..." opens the picker for a post that is NOT yet saved, the implicit Saved row is NOT pre-checked. D-04 / T-50-PICKER-RACE mitigation requires "Saved" to be pre-checked on first open for any post so that single-tap Done saves the post (YouTube "Watch later" pre-check parity).

Purpose: Make CollectionPickerSheet honor the "no refresh" durable rule and restore single-tap-save UX.
Output: Updated CollectionPickerSheet.tsx + a dedicated no-refresh test that mounts the sheet, mutates the service, and asserts the new row appears without remount.
</objective>

<durable_rule>
Capacitor mobile users CANNOT refresh. Every component that reads from a service AND may be on-screen during a mutation MUST subscribe to the service's `*_CHANGED` event and re-read state in a `useEffect`. `useState(() => svc.get())` and `useMemo(() => svc.get(), [...])` initializers fire ONCE; they do not auto-update on service mutations.
