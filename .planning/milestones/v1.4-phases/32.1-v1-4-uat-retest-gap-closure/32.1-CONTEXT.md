<decisions>
## Implementation Decisions

### G1 — Queue cycling + video dedup regression (UAT-31-2 + UAT-31-13)

- D-01: **Both symptoms diagnosed and fixed in the same plan.** UAT-31-2 ("duplicate YouTube videoIds across pull-up cycles") and UAT-31-13 ("'No more posts today' fires before any post is read") are almost certainly the same root cause: the dedup Set in `refillQueue` is not persisting across cycles AND the cycle-counter / queue-exhaustion gate fires too eagerly. They land in one plan, one commit.
- D-02: **Dedup Set must persist across `refillQueue` invocations.** Phase 31-08 added `seenVideoIds` as a local `Set` inside a single `generatePostBatch` call. To prevent duplicates across pulls, it must be promoted to module-scope (or persisted to localStorage scoped to the day). Pick one based on diagnosis output; module-scope is simpler, localStorage is robust to refresh.
- D-03: **Premature exhaustion gate must check `dueConcepts.length > 0`, not the totalGenerated cap alone.** Likely cause: when `getTotalGenerated() >= dailyGenerationCapMultiplier * Math.max(dueConcepts.length, 1)` and `dueConcepts.length` is 0 or low, the cap saturates at 5 posts (1 due concept × 5 multiplier) and the queue exhausts after one batch. Either lower-bound `dueConcepts.length` to a sensible floor (e.g., 3) OR loosen the cap when the user is actively pulling for more.
- D-04: **No new tests required for G1 fix.** `concept-batch-filter.test.mjs` and `daily-generation-cap.test.mjs` already exist (uncommitted on disk). Plan must commit them and verify they catch the regression.

### G2 — Video touch overlay swallows YouTube controls (UAT-31-4)

- D-05: **Reduce overlay scope, not remove it.** Phase 31-09's transparent overlay was added to detect taps for unified video state (start/stop). Removing it entirely re-breaks UAT-31-5 (video stops on swipe-away). Keep gesture detection but use `pointer-events: none` on the inner iframe region OR scope the overlay to only the close-button corner OR use a single-tap event listener that bubbles to YouTube after firing once.
- D-06: **Verify on device, not in unit test.** YouTube iframe pointer behavior is not reproducible in node-test (no real iframe). Acceptance is a device retest line in 31-UAT.md (`retest: pass` for test 4) AND a brief on-device walkthrough by the operator after fix lands.
- D-07: **Tap-to-restart-playback is acceptable as a fallback.** If reaching YouTube's native controls is impossible without breaking gesture detection, the minimum acceptable outcome is: tapping a playing video does nothing (no restart), and a clear stop button is visible. The current "tap restarts playback" behavior is the failure mode.

### G3 — Starter posts retest after Vite base fix (UAT-31-14)

- D-08: **No code change expected — pure retest task.** Commit `8d8fa5fc` (Vite `base: '/'`) unblocks the white-screen regression that prevented 31-10's empty-state guard from running. Plan deliverable is a single retest line in 31-UAT.md after the operator does Settings → Data → Clear All Data → reload on a freshly-built APK.
- D-09: **If retest fails, escalate to fix in the same plan.** If after Fix A is deployed, the user still sees "Nothing new today" instead of 3 starter posts, root cause is in `getDailyPosts` empty-state path (`concept-feed.service.ts:1020-1022`). Add a fix in the same plan; do not split into a separate one.
- D-10: **G3 plan blocks on operator deploying the new APK.** Plan task is a `checkpoint:human-verify` that requires the operator to confirm they've run `npm run build && npx cap sync android && [reinstall]` before retesting. Without that, the white-screen bug persists regardless of code state.

### G4 — Starter posts persist after first view (NEW)

- D-11: **Persist starter posts to the daily cache on first render.** Current behavior: `STARTER_POSTS` are returned by `getDailyPosts` only when `questions.length === 0` (`concept-feed.service.ts:1021`). Once the user taps a starter post and it potentially mutates state (e.g., creates a question via the "Welcome" CTA), the empty-state path is no longer taken on the next /home visit, and starter posts vanish. Fix: on first cold-start render where `STARTER_POSTS` are returned, also persist them to the daily cache via `saveCache({ date, fingerprint, posts: STARTER_POSTS, connectionCards: [] })` so subsequent visits return them from cache regardless of `questions.length`.
- D-12: **Decay condition: starter posts disappear once user has 3+ organic posts.** Don't keep starter posts forever — once the user has accumulated enough real content (3 organic concept posts in cache), the cold-start fallback is no longer needed and starter posts should be filtered out (or replaced) on the next cache load. Threshold: `cached.posts.filter(p => !STARTER_POST_IDS.has(p.id)).length >= 3` → drop starters.
- D-13: **Starter posts must navigate to a real PostDetailScreen, not a stub.** When tapping a starter post, the user should land on a normal-looking PostDetailScreen with the starter content visible. Verify this works post-fix; if PostDetailScreen errors on starter post IDs (e.g., `welcome-1`), add a guard in PostDetailScreen's loader to render starter content directly.

### G5 — Clear All Data auto-reload (NEW from 32.1-01 retest 2026-04-19)

- D-19: **Replace `window.location.reload()` with `window.location.assign('/home')`.** The reload call at `SettingsDataScreen.tsx:68` is a silent no-op in Capacitor's WebView on Android (operator confirmed: stayed on /settings/data after Clear All Data, had to manually navigate to /home). `assign('/home')` does both: navigates to the home route AND triggers a fresh page load. This also semantically aligns with user intent — clearing all data should drop them back to the fresh-start experience.
- D-20: **No regression test required.** Pure UI behavior change at a single call site; manual operator retest after deploy is sufficient acceptance. The G3 plan already gates on `apk-deployed` so the retest cycle naturally validates G5 too. Acceptance criterion: after Clear All Data, the app navigates to /home (not /settings/data) without manual intervention.

### Cross-cutting

- D-14: **One plan per gap, five plans total** (was four; updated 2026-04-19 after 32.1-01 retest surfaced G5). No combining G1+G3 or G2+G4 even though they touch overlapping files. Atomic commits, atomic plans, easier to revert if one regresses.
- D-15: **Wave structure:** G3 first (it's a pure retest, fast), then G1 + G2 + G4 + G5 in parallel (they touch different files). Dependencies: G3 doesn't block anything; G1/G2/G4/G5 are independent.
- D-16: **All commits land on `main`. No phase branch.** Same convention as Phase 32.
- D-17: **Each plan must update `31-UAT.md` with a `retest:` row** for the corresponding test where one exists. G3 closes UAT-31-14, G1 closes UAT-31-2 and UAT-31-13, G2 closes UAT-31-4. G4 and G5 have no Phase 31 UAT counterpart — record in their respective `32.1-NN-SUMMARY.md` instead.
- D-18: **Tests for G1 and G4 are MANDATORY.** Both gaps are regressions of code that was supposed to be fixed; tests prevent re-regression. G1 has existing untracked tests on disk that must be committed. G4 needs a new test asserting starter posts persist after the first cache write. G5 is exempt per D-20 (single-line UI behavior change, manual retest sufficient).

## Specific Ideas

### G1 — concrete diagnosis steps to run before fix
1. Add a `console.log('[refillQueue] cycle', cycleNumber, 'totalGenerated', totalGenerated, 'maxPosts', maxPosts, 'dueConcepts', dueConcepts.length)` to `refillQueue` and observe on device through several pull cycles
2. Check whether `seenVideoIds` is being recreated each call (likely yes — it's local) — if so, promote to module-scope or pass through as a parameter that reads from a persistent store
3. Check `dailyGenerationCapMultiplier` setting on user's device — if 1, that's the cap source

### G2 — three candidate fix patterns (pick one based on iframe DOM inspection)
- (a) `pointer-events: none` on the iframe wrapper, `pointer-events: auto` only on the close-button overlay
- (b) Single-fire tap detector that fires once then removes itself, letting subsequent taps pass through
- (c) Two-zone overlay: top 80% transparent passthrough to iframe, bottom 20% (or close-button corner) intercepts

### G3 — retest dialogue with operator
"Settings → Data → Clear All Data → confirm. Wait for app to reload. Do you see (a) 3 starter posts, (b) 'Nothing new today', or (c) white screen still?"
