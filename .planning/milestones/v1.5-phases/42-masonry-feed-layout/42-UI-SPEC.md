## Spacing Scale

Project's existing 4-grid scale (CSS vars in `app/src/index.css:98-104`). MasonryFeed and the celebration card MUST consume these tokens — no new pixel values introduced.

| Token | Value | Usage in this phase |
|-------|-------|---------------------|
| `--space-xs` | 4px | Icon-to-label gap inside celebration action rows |
| `--space-sm` | 8px | Inter-column horizontal gap (between left/right column buckets) |
| `--space-md` | 12px | **Inter-tile vertical gap** within each column (matches existing `gap: 12px` on `InfoFlow.tsx:841` flex container — preserves visual rhythm). See Exceptions below. |
| `--space-lg` | 16px | Celebration card internal padding (vertical between sections); MasonryFeed wrapper horizontal padding (matches HomeScreen `paddingLeft/Right: '16px'`) |
| `--space-xl` | 20px | Celebration card outer padding around the vine SVG. See Exceptions below. |
| `--space-2xl` | 24px | Celebration card top-margin from last masonry tile (matches `--section-gap`) |
| `--section-gap` | 24px | Vertical gap between celebration card and the masonry grid above it |

**Inter-tile gap (within column):** 12px (`--space-md`) — matches the existing single-column flex gap; tile rhythm is preserved when columns split.

**Inter-column gap (horizontal):** 8px (`--space-sm`) — tighter than vertical gap so the masonry reads as a coherent grid (Pinterest/Rednote convention: horizontal gap < vertical gap).

**Celebration card spans both columns** with `0` horizontal margin (full-bleed inside the MasonryFeed wrapper's existing horizontal padding).

**Tap target minimum:** 44px height for celebration card action rows (project convention; matches PlannerScreen suggested-moves rows).

**Exceptions:**

- **12px (`--space-md`):** pre-existing project token from `app/src/index.css:98-104`; reused as inter-tile vertical gap to match the existing `gap: 12px` on `InfoFlow.tsx:841` (preserves existing visual rhythm; introducing 8px or 16px would disrupt the established feed cadence).
- **20px (`--space-xl`):** pre-existing project token from `app/src/index.css:98-104`; reused as celebration card outer padding around the vine SVG (chosen to balance the 88×88 SVG viewport with surrounding text without pushing the card to feel oversized at 24px or undersized at 16px).
- **88×88 SVG viewBox:** intrinsic to the vine illustration asset, not a layout token.

## Typography

Project has no formal type-scale token system (existing screens hardcode rem/px values inline). For Phase 42, the celebration card adopts a **4-size hierarchy** consistent with the existing `MilestoneCard` (`InfoFlow.tsx:700-728`) and HomeScreen empty-states (`HomeScreen.tsx:743-748`), bound to a **2-weight scheme**.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Display (celebration title) | 1.25rem (20px) | 700 (bold) | 1.3 | "Vine in bloom" headline at top of celebration card |
| Body (celebration body / fallback copy / section header) | 0.95rem (~15px) | 400 (regular) | 1.5 | Suggestions header text ("Tomorrow's vine tending"), static fallback prose ("Your vine is fully healthy...") |
| Label (action row title) | 0.9rem (~14px) | 400 (regular) | 1.4 | Each tappable suggestion row's primary label (e.g., "Heal 'Spaced Repetition'") |
| CTA (Open Planner) | 0.9rem (~14px) | 700 (bold) | 1.4 | "Open Planner" link at the bottom of the celebration card — bold to mirror the headline weight and signal primary action |
| Caption (action row sub-label) | 0.8rem (~13px) | 400 (regular) | 1.4 | Per-row state badge ("dying", "dead") |

**Weight discipline (max 2):** Only **700 (bold)** and **400 (regular)** appear anywhere in this phase's typography. 700 is reserved for the headline ("Vine in bloom") and the primary CTA ("Open Planner"); 400 is used for everything else (section header, action row labels, fallback body, captions). Weights 500 and 600 are explicitly NOT used in Phase 42.

**Why these specific sizes:** All 4 sizes already exist in the codebase (`MilestoneCard.headline`, `HomeScreen.emptyTitle`, `PlannerScreen.deadRow.label`, `PlannerScreen.deadRow.subtitle`). Phase 42 introduces zero new font sizes — the celebration card composes existing scale members.

**Color:** Title and CTA in `var(--primary-40)`; action row labels in `var(--foreground)`; body, fallback prose, and captions in `var(--muted-foreground)`.

**MasonryFeed itself adds no new typography** — it is a layout container; child cards (`MemoizedConceptCard`, `ConnectionCard`, `MilestoneCard`) own their own type contracts unchanged.

## Animation Contract

framer-motion `Variants` for tile entrance (exact shape — planner consumes verbatim):

```typescript
const tileEnterVariants: Variants = {
  hidden: { opacity: 0, y: 8 },
  visible: { opacity: 1, y: 0 },
};

const tileEnterTransition = {
  duration: 0.25,                    // D-05: 250ms
  ease: [0.25, 0.1, 0.25, 1] as const, // ease-out cubic-bezier (Material standard)
};

const containerStaggerVariants: Variants = {
  hidden: {},
  visible: {
    transition: {
      staggerChildren: 0.04,         // D-05: 40ms stagger
    },
  },
};
```

**Where applied (D-03 invariant):**
- `<motion.div>` wraps each leaf tile in MasonryFeed's items.map loop, ONLY when `newPostIds.has(itemId)` is true (D-04).
- Tiles NOT in `newPostIds` render as plain `<div>` (no motion wrapper, no animation).
- `<motion.div variants={tileEnterVariants}>` is the leaf; `<motion.div variants={containerStaggerVariants}>` may wrap the items.map IF the items.map is freshly rendered (alternative: use framer's `AnimatePresence` with `initial` prop on each motion leaf).
- NEVER wrap MasonryFeed root, column wrappers, or HomeScreen scroll container in `<motion.div>` (CLAUDE.md Header positioning rule + ROADMAP SC-4 invariant).

**Celebration card entrance:**

```typescript
const celebrationVariants: Variants = {
  hidden: { opacity: 0, y: 12 },
  visible: { opacity: 1, y: 0 },
};
const celebrationTransition = {
  duration: 0.22,                  // 220ms — slightly faster than tiles (avoids feeling sluggish)
  ease: [0.25, 0.1, 0.25, 1] as const,
};
```

**SVG bloom path draw:**

```typescript
const bloomPathVariants: Variants = {
  hidden: { pathLength: 0, opacity: 0 },
  visible: { pathLength: 1, opacity: 1 },
};
const bloomPathTransition = {
  duration: 0.6,                   // 600ms — gentle bloom unfurl
  ease: [0.4, 0, 0.2, 1] as const, // standard easing
  delay: 0.15,                     // 150ms after card fades in
};
```

**Reduced-motion compliance:** All `<motion.*>` elements use framer-motion's default `useReducedMotion` honoring — animations become instant when the OS setting is on. No additional code required. Celebration card always renders (no blank state).

## Layout Algorithm: Height-Accumulating Split (D-02)

Verbatim from CONTEXT.md D-02. Spec for planner consumption:

**State:**
- `columnHeightsRef = useRef<[number, number]>([0, 0])` — [col-A, col-B] accumulated heights in px.
- `tileColumnAssignmentsRef = useRef<Map<string, 0 | 1>>(new Map())` — itemId → column index, append-only.
- `tileRefsMap = useRef<Map<string, HTMLDivElement | null>>(new Map())` — for `useLayoutEffect` height measurement.

**Algorithm (per render):**
1. `useLayoutEffect` runs after render. For each item in `items[]`:
   - If `tileColumnAssignmentsRef.has(itemId)` → reuse existing assignment (immutability invariant).
   - Else → assign to `columnHeightsRef.current[0] <= columnHeightsRef.current[1] ? 0 : 1` (tie → col 0); record in map.
2. After all assignments, measure each tile's `clientHeight` and update `columnHeightsRef.current[col] += height + 12` (gap inclusion).
3. Re-render not needed for the algorithm itself — assignments are stable; tile placement is via two filtered render passes (col-A items, col-B items).

**Render shape:**

```jsx
<div style={{ display: 'flex', gap: '8px' }}>
  <div style={{ flex: 1, display: 'flex', flexDirection: 'column', gap: '12px' }}>
    {items.filter((item) => assignmentMap.get(getId(item)) === 0).map(renderTile)}
  </div>
  <div style={{ flex: 1, display: 'flex', flexDirection: 'column', gap: '12px' }}>
    {items.filter((item) => assignmentMap.get(getId(item)) === 1).map(renderTile)}
  </div>
</div>

{/* Celebration card — sibling AFTER the column flex container */}
{allExplored && layout.nodes.length > 0 && (
  <div style={{ marginTop: '24px' }}>
    <VineBloomCard ... />
  </div>
)}
```

**No `transform`, `will-change`, `filter`, `contain`, or `perspective`** on either column wrapper or the MasonryFeed root (CLAUDE.md "Header positioning" load-bearing rule).

**No `position: fixed`** on column wrappers (CLAUDE.md Phase 32.1 lesson 3 — `position:fixed` + `overflow:auto` + Android WebView is a recurring bug class).

**Tile measurement:** `useLayoutEffect` post-render measurement on items length change. `ResizeObserver` is **not** required for v1 — recommended only if UAT shows visible column desync from async image loads (CONTEXT.md "ship the simpler thing first").
